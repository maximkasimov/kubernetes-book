# Kubernetes: подробный учебный материал — Pod и жизненный цикл контейнеров: минимальная единица размещения

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## Pod и общая среда

Pod — единица планирования: его контейнеры находятся на одной node, разделяют IP-адрес, network namespace и при необходимости volumes. Поэтому контейнеры одного Pod общаются через localhost, но не могут использовать одинаковый порт. Один основной контейнер на Pod проще масштабировать и диагностировать; sidecar оправдан только общей сетью, файлами или жизненным циклом.

Изучайте ресурс через Kubernetes API, а не как отдельный YAML-файл. API server хранит желаемое состояние, а контроллеры асинхронно приводят кластер к нему. Поэтому момент успешного apply и момент готовности приложения различны.

Для любого объекта различайте spec и status. spec задаёт намерение, status и conditions сообщают результат работы контроллеров. Conditions с полями type, status, reason и message обычно точнее краткой колонки kubectl get.

События (Events) полезны как временная диагностическая подсказка. Их нельзя считать долговременным аудитом: сохраняйте важные факты в системе логирования и runbook. Всегда проверяйте ownerReferences, чтобы понять, какой контроллер создал объект.

Безопасная диагностика идёт сверху вниз: пользовательский результат, ресурс верхнего уровня, его потомки, node/runtime и внешняя зависимость. Массовое удаление Pod или выдача cluster-admin может скрыть причину, а не устранить её.

## От назначения к процессу

Scheduler выбирает node, kubelet получает назначение, создаёт sandbox через CRI, CNI подключает сеть, runtime загружает образы и монтируются volumes. Затем последовательно работают init containers, и только после их успеха запускаются обычные контейнеры. ContainerCreating — не причина, а стадия: Events различат image pull, mount и CNI.

## Фазы и состояния

Pod phase Pending, Running, Succeeded, Failed или Unknown — грубая сводка. Контейнер имеет Waiting, Running либо Terminated с reason и exit code. Conditions PodScheduled, Initialized, ContainersReady и Ready показывают границу, на которой остановился путь. Running не означает, что Pod принимает трафик: для этого обычно требуется Ready.

## Завершение и диагностика

При удалении kubelet выполняет preStop, посылает SIGTERM и ждёт terminationGracePeriodSeconds, затем использует SIGKILL. Приложение должно прекратить новые запросы и завершить текущие за этот срок. CrashLoopBackOff означает задержку повторных рестартов; смотрите previous logs, exit code, probes, command и mounts. Ephemeral containers позволяют диагностировать минимальный образ, но требуют строгого RBAC.

## Официальные источники для темы

Проба не должна зависеть от endpoint, который сам проходит через тот же Service/Gateway и создаёт циклический отказ. Отделяйте локальную готовность процесса от проверок удалённых бизнес-зависимостей. Для каждой probe фиксируйте timeout, period, threshold и ожидаемую причину рестарта/исключения endpoint.

Readiness снимает endpoint из Service, однако уже открытые соединения могут продолжать идти, а внешние балансировщики имеют собственные health-check/connection draining. Правильная остановка координирует readiness, preStop, SIGTERM, grace period и timeout Gateway. Если terminationGracePeriod меньше времени обработки запроса или закрытия базы, будет потеря транзакций.

## Углубление: probes и termination в реальном rollout

restartPolicy Always обычно применяется workload-контроллерами; OnFailure/Never характерны для Job. Перезапуск контейнера не пересоздаёт Pod и не меняет Pod IP, но полное пересоздание контроллером создаёт новый Pod/UID. Приложение должно корректно различать временный рестарт процесса и потерю всей реплики.

Ресурсные requests/limits относятся к каждому контейнеру, а scheduler суммирует их для Pod с учётом init containers. Init container с большим request может определять возможность размещения, даже если после запуска он завершится. Это нужно учитывать при подготовительных миграциях и bootstrap.

## Углубление: requests, limits и container lifecycle

Образ контейнера фиксируйте digest либо контролируемым неизменяемым тегом. imagePullPolicy не гарантирует воспроизводимость при изменяемом теге: разные nodes могут получить разные байты. Для production полезно записывать imageID/digest в статусе и связывать его с артефактом CI.

Kubelet — агент node, который наблюдает назначенные Pod и через Container Runtime Interface (CRI) просит runtime создать sandbox и контейнеры. Runtime управляет образами и процессами, но не принимает решения scheduler. Ошибка image pull, sandbox, mount volume и запуск process имеет разные события и логи; общий статус Pending/ContainerCreating не заменяет эту классификацию.
