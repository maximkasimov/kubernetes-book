# Kubernetes: подробный учебный материал — CNI: контракт, IPAM и подключение Pod к сети

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## CNI в жизненном цикле Pod

Container Network Interface (CNI) — спецификация и набор плагинов, через которые runtime/kubelet подключает sandbox Pod к сети. Kubernetes не реализует dataplane сам: он вызывает выбранный CNI с данными о Pod, network namespace и конфигурации. Плагин создаёт интерфейс, назначает IP-адрес, маршруты и при необходимости правила policy/туннеля. Ошибка на этом этапе оставляет Pod в ContainerCreating.

Изучайте ресурс через Kubernetes API, а не как отдельный YAML-файл. API server хранит желаемое состояние, а контроллеры асинхронно приводят кластер к нему. Поэтому момент успешного apply и момент готовности приложения различны.

Для любого объекта различайте spec и status. spec задаёт намерение, status и conditions сообщают результат работы контроллеров. Conditions с полями type, status, reason и message обычно точнее краткой колонки kubectl get.

События (Events) полезны как временная диагностическая подсказка. Их нельзя считать долговременным аудитом: сохраняйте важные факты в системе логирования и runbook. Всегда проверяйте ownerReferences, чтобы понять, какой контроллер создал объект.

Безопасная диагностика идёт сверху вниз: пользовательский результат, ресурс верхнего уровня, его потомки, node/runtime и внешняя зависимость. Массовое удаление Pod или выдача cluster-admin может скрыть причину, а не устранить её.

## Что получает и возвращает плагин

CNI вызывается для ADD при создании, DEL при удалении и CHECK, если это поддержано. Плагин получает JSON-конфигурацию, имя контейнера, network namespace и путь к CNI-конфигурации. Результат содержит IP, routes, DNS-информацию и интерфейсы. Цепочка плагинов может сначала создать veth, затем назначить адрес через IPAM и добавить port mapping или policy. Все шаги обязаны корректно освобождать состояние при DEL.

## IPAM и адресный план

IP Address Management (IPAM) выделяет Pod адрес из диапазона. Он может быть host-local, централизованным или интегрированным в CNI. Адресный план должен учитывать максимальное число Pods на node, node CIDR, overlay/underlay и отсутствие пересечений с сетью nodes, VPN и корпоративными подсетями. Исчерпание IP часто проявляется только на новых Pods, тогда как старые остаются здоровыми.

## veth и network namespace

Обычно CNI создаёт veth-пару: один конец находится в namespace Pod как eth0, другой — на node и подключён к bridge, маршруту или eBPF dataplane. Контейнеры одного Pod видят общий eth0, но отдельные Pods изолированы namespace. Понимание этой границы объясняет, почему tcpdump на node и внутри Pod показывает разные интерфейсы и почему localhost не пересекает Pod.

## Безопасность и эксплуатация

CNI-agent обычно работает с высокими привилегиями на node, потому что меняет network namespace, routes и dataplane. Его RBAC, образ, обновление и socket runtime — часть критичной поверхности кластера. План обновления CNI требует проверки совместимости Kubernetes, kernel, MTU, CRD и rollback. Нельзя считать сетевой плагин «обычным приложением»: ошибка обновления затрагивает каждый новый Pod.

## Официальные источники для темы

При подозрении на утечку не удаляйте записи IPAM массово. Сравните фактические Pods/UID на node с allocation state плагина, завершённые sandbox, kubelet/runtime логи и время последнего изменения. Ручная выдача уже используемого адреса создаёт конфликт, который выглядит как случайная недоступность и может затронуть другой workload.

Расчёт ёмкости IP — не только размер cluster CIDR. Считайте максимум Pods на node, системные DaemonSet, запас на rollout maxSurge, ограничения выбранного CNI, зарезервированные адреса и topology. В некоторых реализациях адреса выделяются блоками node, поэтому свободные IP в общем пуле не означают, что конкретная node может принять Pod. Метрика utilisation IPAM и alert на приближение к пределу предотвращают аварию лучше, чем ручная чистка.

## Углубление: IPAM, capacity и утечки

CNI не выбирает node и не назначает Pod: это уже сделал scheduler. Он также не создаёт Service и не делает HTTP-маршрутизацию. Чёткое разделение предотвращает неверную диагностику: Pending с FailedScheduling не является CNI-сбоем; ContainerCreating с ошибкой network setup — уже CNI/runtime граница.

Когда kubelet создаёт Pod sandbox, runtime создаёт сетевое пространство имён и передаёт его путь CNI. Главный плагин получает ADD: создаёт veth, один конец переносит в namespace, настраивает адрес и default route; второй остаётся на node. IPAM выделяет адрес и возвращает результат; затем dataplane-плагин добавляет маршрут, tunnel endpoint, eBPF endpoint или firewall state. Только после успеха сети runtime продолжает запуск контейнеров. При частичной ошибке требуется корректный DEL: иначе остаётся утёкший IP, veth или запись в endpoint database.
