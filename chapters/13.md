# Kubernetes: подробный учебный материал — Service и внутренний доступ: стабильная точка для Pod

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## Почему не Pod IP

Pod эфемерен: controller пересоздаёт его, IP меняется, node может стать недоступной. Service отделяет потребителя от набора реплик, даёт DNS-имя и виртуальный адрес. Service selector выбирает Pods; контроллер создаёт EndpointSlice. Это два отдельных шага: наличие Service не означает наличие backend.

Изучайте ресурс через Kubernetes API, а не как отдельный YAML-файл. API server хранит желаемое состояние, а контроллеры асинхронно приводят кластер к нему. Поэтому момент успешного apply и момент готовности приложения различны.

Для любого объекта различайте spec и status. spec задаёт намерение, status и conditions сообщают результат работы контроллеров. Conditions с полями type, status, reason и message обычно точнее краткой колонки kubectl get.

События (Events) полезны как временная диагностическая подсказка. Их нельзя считать долговременным аудитом: сохраняйте важные факты в системе логирования и runbook. Всегда проверяйте ownerReferences, чтобы понять, какой контроллер создал объект.

Безопасная диагностика идёт сверху вниз: пользовательский результат, ресурс верхнего уровня, его потомки, node/runtime и внешняя зависимость. Массовое удаление Pod или выдача cluster-admin может скрыть причину, а не устранить её.

## Типы Service

ClusterIP доступен в кластере. NodePort резервирует порт на nodes и обычно используется как строительный блок, а не финальный публичный интерфейс. LoadBalancer запрашивает внешний балансировщик у интеграции облака/MetalLB/другого контроллера. ExternalName возвращает DNS CNAME и не создаёт proxy; его поведение зависит от DNS-клиента.

## EndpointSlice и готовность

EndpointSlice масштабируемо представляет backends и их readiness. Обычно Service направляет трафик только на ready endpoints. Если Pod работает, но readiness false, его не будет в обычном наборе. При расследовании сравните selector, labels Pod, EndpointSlice, ports и conditions; не проверяйте Service только curl с одной node.

## Официальные источники для темы

Сначала запросите EndpointSlice и убедитесь, что labels/ports совпали. Затем из диагностического Pod того же namespace резолвите Service, проверяйте TCP targetPort и прикладной протокол. Если тест из Pod проходит, а через Gateway нет, проблема выше Service. Если Pod IP работает, а Service нет, сравнивайте Service translation, selector и endpoints.

## Углубление: корректный тест

Headless Service с clusterIP: None не даёт виртуальный IP и возвращает DNS-записи endpoints. Он нужен там, где клиент умеет работать с отдельными репликами, например StatefulSet. Это не значит, что можно подключаться к произвольному Pod IP без проверки его готовности и жизненного цикла.

Service controller строит EndpointSlice из Pods, подходящих selector. Dataplane выбирает endpoint по реализации kube-proxy/eBPF. Endpoint с terminating/ready conditions может обрабатываться по-особому, поэтому rolling update требует сочетания readiness, grace period и draining. Service не устанавливает соединение сам: это виртуальная точка выбора backend, а TCP/HTTP поведение остаётся у клиента, gateway и приложения.

В учебном кластере создайте минимальный пример рассматриваемой сущности. До apply выполните server-side dry-run и diff. После применения сохраните YAML, describe, events и логи.
