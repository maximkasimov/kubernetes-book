# Kubernetes: подробный учебный материал — Calico и NetworkPolicy: изоляция и управляемое разрешение трафика

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## Calico как CNI и policy engine

Calico может давать CNI-подключение, маршрутизацию через BGP или overlay и применять policy. Конкретный dataplane и возможности зависят от выбранной конфигурации, поэтому нельзя переносить пример «Calico» без проверки режима. Сначала фиксируйте CIDR, encapsulation, BGP peers, MTU и способ реализации policy.

Изучайте ресурс через Kubernetes API, а не как отдельный YAML-файл. API server хранит желаемое состояние, а контроллеры асинхронно приводят кластер к нему. Поэтому момент успешного apply и момент готовности приложения различны.

Для любого объекта различайте spec и status. spec задаёт намерение, status и conditions сообщают результат работы контроллеров. Conditions с полями type, status, reason и message обычно точнее краткой колонки kubectl get.

События (Events) полезны как временная диагностическая подсказка. Их нельзя считать долговременным аудитом: сохраняйте важные факты в системе логирования и runbook. Всегда проверяйте ownerReferences, чтобы понять, какой контроллер создал объект.

Безопасная диагностика идёт сверху вниз: пользовательский результат, ресурс верхнего уровня, его потомки, node/runtime и внешняя зависимость. Массовое удаление Pod или выдача cluster-admin может скрыть причину, а не устранить её.

## Семантика Kubernetes NetworkPolicy

NetworkPolicy выбирает Pods по podSelector и разрешает ingress/egress для выбранных Pods. Когда Pod выбран политикой данного направления, действует модель allowlist: разрешён только явно совпавший трафик; неселектированные Pods остаются вне этой изоляции. Политика не включает CNI-плагин сама: enforcement должен поддерживаться установленным CNI.

## Selectors и пересечение правил

podSelector выбирает источник/получатель внутри namespace; namespaceSelector расширяет область; ipBlock описывает CIDR. Несколько политик складываются как объединение разрешений, а не как последовательный deny. Поэтому нельзя «перекрыть» широкое разрешение более узкой запретительной NetworkPolicy: в стандартной семантике нет явного deny.

## DNS и egress

Default deny egress без исключения для DNS ломает резолвинг, часто проявляясь как ошибки приложений. Разрешение DNS должно быть точным: target DNS service/namespace/port, с пониманием того, как конкретный CNI сопоставляет IP и policy. Затем добавляйте только нужные базы, внешние CIDR или Service. Не используйте 0.0.0.0/0 как постоянный обход проблемы.

## Углубление: Calico control plane

Политика применяется к Pod по labels в момент передачи трафика. Изменение labels deployment может не только поменять выбор Service, но и изменить сетевые права. Согласуйте набор security-labels, защитите их admission-политикой и не используйте свободные произвольные labels как security-boundary.

Стандартная Kubernetes NetworkPolicy имеет аддитивную семантику: если хотя бы одно правило разрешает поток, он допускается. Политики не образуют порядок «последнее правило победило», и стандартный API не имеет deny-правила, которое переопределит ранний allow. Поэтому широкий namespaceSelector или пустой podSelector должны проходить особенно строгое ревью.
