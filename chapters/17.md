# Kubernetes: подробный учебный материал — Gateway API: HTTP-маршрутизация, TLS и политики

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## HTTPRoute

HTTPRoute сопоставляет hostnames и matches: path, headers, query parameters, method. Rules задают backendRefs и filters. Порядок и допустимые комбинации определяет спецификация/контроллер; не переносите предположение из NGINX Ingress. BackendRef должен указывать существующий Service и разрешённый port, иначе ResolvedRefs/status даст причину.

Изучайте ресурс через Kubernetes API, а не как отдельный YAML-файл. API server хранит желаемое состояние, а контроллеры асинхронно приводят кластер к нему. Поэтому момент успешного apply и момент готовности приложения различны.

Для любого объекта различайте spec и status. spec задаёт намерение, status и conditions сообщают результат работы контроллеров. Conditions с полями type, status, reason и message обычно точнее краткой колонки kubectl get.

События (Events) полезны как временная диагностическая подсказка. Их нельзя считать долговременным аудитом: сохраняйте важные факты в системе логирования и runbook. Всегда проверяйте ownerReferences, чтобы понять, какой контроллер создал объект.

Безопасная диагностика идёт сверху вниз: пользовательский результат, ресурс верхнего уровня, его потомки, node/runtime и внешняя зависимость. Массовое удаление Pod или выдача cluster-admin может скрыть причину, а не устранить её.

## Разделение TLS

TLS termination означает: Gateway принимает TLS, имеет сертификат Secret/certificateRef и передаёт HTTP backend. TLS passthrough оставляет TLS до backend и ограничивает L7-наблюдение/маршрутизацию. Выбор зависит от trust boundary, mTLS, требований приложения и возможностей controller. Сертификат — секретный материал: доступ и namespace references должны быть ограничены.

## Cross-namespace ссылки

ReferenceGrant позволяет владельцу целевого namespace явно разрешить ссылку от другого namespace на Service/Secret/другой ресурс. Это предотвращает ситуацию, когда один tenant публикует чужой backend. Отсутствие ReferenceGrant — корректный отказ, а не дефект Gateway; читайте ResolvedRefs и сообщения status.

## Официальные источники для темы

TLS termination делает сертификат доступным dataplane Gateway; это требует минимального доступа и ротации. ReferenceGrant явно открывает cross-namespace ссылку, например Route на backend Service. Его отсутствие — защита multi-tenancy, которую нельзя обходить выдачей широкого RBAC application team. Для сертификатной автоматизации проверьте issuer, secret lifecycle и policy допуска secret refs.

## Углубление: TLS и cross-namespace

Filters меняют запрос/ответ на edge. Redirect и rewrite меняют контракт с клиентом, поэтому тестируйте canonical URL, заголовки, cookies и caching. Не превращайте Gateway в место скрытой прикладной логики: сложные auth/retry/transformation требуют явной платформенной политики и owners.

HTTPRoute может сопоставлять hostname, path, header, query и method. Правило должно быть минимальным и однозначным: слишком общий path или hostname перехватывает трафик другого приложения. backendRefs задают веса для traffic split; сумма и поддержка правил определяются спецификацией/реализацией. При ошибке смотрите route status, backend Service/port и EndpointSlice, прежде чем отлаживать HTTP-код приложения.
