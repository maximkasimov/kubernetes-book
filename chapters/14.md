# Kubernetes: подробный учебный материал — NetworkPolicy в production: модель угроз, rollout и диагностика

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## Граница и модель угроз

NetworkPolicy ограничивает сетевые связи Pods, но не заменяет RBAC, TLS, identity приложения, image security или secrets. Сформулируйте активы и разрешённые потоки: ingress controller к backend, backend к database, Pod к DNS, monitoring к metrics. Политика должна быть проверяемым выражением этой модели, а не копией случайных CIDR.

## Default deny

Default deny выбирает все Pods namespace и включает PolicyTypes Ingress/Egress без разрешений. Это мощный базовый слой, но опасен без инвентаризации: ломаются DNS, image/identity endpoint, metrics, внешние API и межnamespace зависимости. Сначала подготовьте точные allow policies и наблюдение, затем включайте deny по ограниченной области.

## Идентификация зависимостей

Предпочитайте labels и namespace selectors для внутрикластерных сервисов, когда они отражают роль. ipBlock оставляйте для внешних сетей с устойчивыми CIDR. Не используйте IP Service как замену selector без понимания CNI: policy применяется к трафику на разных стадиях dataplane. Для database важны protocol/port и направление, а не только адрес.

## Проверка эффекта

Тест должен запускаться от того же ServiceAccount/namespace и идти из Pod, где реально возникает поток. Проверяйте DNS отдельно от TCP и прикладного HTTP/gRPC. Flow-observability CNI помогает найти verdict, но итоговый тест нужен для подтверждения. Сохраняйте baseline разрешённых потоков до rollout.

## Инциденты и откат

При массовом deny первым делом ограничьте blast radius и верните проверенный набор предыдущих манифестов. Не открывайте весь egress навсегда «до завтра»: после восстановления зафиксируйте недостающий поток и добавьте точное правило. Изменения policy проходят ревью так же, как firewall production-сети.

## Официальные источники для темы

NetworkPolicy не ограничивает hostNetwork Pods таким же образом во всех реализациях и не заменяет perimeter firewall. Трафик к API server, node-local сервисам, DNS и metadata endpoints требует отдельных решений. Для egress во внешнюю сеть уточните, видит ли policy исходный IP до или после NAT в вашем CNI, и избегайте предположений по чужой документации.

## Углубление: границы NetworkPolicy

Тесты должны содержать позитивную и негативную проверку: допустимый backend доступен, недопустимый backend недоступен. Только позитивный curl не доказывает изоляцию. Фиксируйте protocol/port, потому что разрешение TCP 443 не разрешает UDP 53 и наоборот.

Переход начинайте с режима наблюдения CNI и карты потоков за репрезентативный период. Для namespace создайте сначала allow DNS и все известные обязательные зависимости, протестируйте их от реальных Pods, затем добавляйте default deny в окно с rollback. Изменение namespace с сотнями workload не должно быть первой проверкой модели; начните с малого изолированного сервиса.
