# Kubernetes: подробный учебный материал — Итоговый production-проект: проектирование Kubernetes-платформы и приложения

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## Исходная архитектура

Спроектируйте сервис с несколькими репликами, Service, storage при необходимости, CNI и default-deny NetworkPolicy, Gateway API edge и observability. Сначала опишите зависимости и потоки: клиент → Gateway → Service → workload → database/DNS/внешние API. Для каждой стрелки определите protocol, identity, TLS, timeout, policy и измерение.

Изучайте ресурс через Kubernetes API, а не как отдельный YAML-файл. API server хранит желаемое состояние, а контроллеры асинхронно приводят кластер к нему. Поэтому момент успешного apply и момент готовности приложения различны.

Для любого объекта различайте spec и status. spec задаёт намерение, status и conditions сообщают результат работы контроллеров. Conditions с полями type, status, reason и message обычно точнее краткой колонки kubectl get.

События (Events) полезны как временная диагностическая подсказка. Их нельзя считать долговременным аудитом: сохраняйте важные факты в системе логирования и runbook. Всегда проверяйте ownerReferences, чтобы понять, какой контроллер создал объект.

Безопасная диагностика идёт сверху вниз: пользовательский результат, ресурс верхнего уровня, его потомки, node/runtime и внешняя зависимость. Массовое удаление Pod или выдача cluster-admin может скрыть причину, а не устранить её.

## Приёмка

Приёмка завершена, когда документация описывает topology, CIDR, CNI/Gateway implementation, flow policies, backup/restore, upgrade и incident runbook; pipeline воспроизводит deployment; тесты подтверждают разрешённый и запрещённый трафик, TLS и rollback. Production-ready — наблюдаемое и восстанавливаемое состояние, не просто зелёный apply.

## Официальные источники для темы

Решение готово, когда все разрешения least privilege, labels/security contracts защищены, policy имеет позитивные и негативные тесты, Git history воспроизводит состояние, а runbook содержит команды и ожидаемые признаки. Зеленый CI без проверки runtime, CNI/Gateway и restore — недостаточное доказательство production readiness.

## Углубление: итоговая оценка

Проверьте отказ: drain одной node, недоступность backend replica, ротация certificate, rollback deployment и восстановление данных на учебном контуре. Цель не в том, чтобы всё выдерживало без деградации, а в том, чтобы ожидаемая деградация была измерима и имела безопасный путь восстановления.

Приёмочный тест создаёт новый Pod с минимальным ServiceAccount, проверяет DNS, разрешённый Service, запрещённый Service, внешнее egress по policy, PVC mount, HTTPRoute/Gateway TLS и лог/метрику. Тест должен выполняться в CI/периодически, а не только вручную после установки.

## Углубление: сквозная проверка

Для каждого shared component определите собственника, SLO, capacity, upgrade policy, backup/restore и incident escalation. Особенно критичны CNI, CoreDNS, Gateway controller, certificate issuer, CSI и observability: приложение зависит от них, даже если его Deployment выглядит простым.

Итоговый проект начинается с документа topology: namespaces, CIDR, nodes/zones, CNI mode, Service/Pod ranges, Gateway implementation, storage classes и внешние зависимости. Затем создаётся таблица потоков с source, destination, DNS name/IP, protocol/port, TLS termination, identity, NetworkPolicy и владельцем. Эта таблица превращает «сеть должна работать» в проверяемый контракт.

## Домашняя практика

В учебном кластере создайте минимальный пример рассматриваемой сущности. До apply выполните server-side dry-run и diff. После применения сохраните YAML, describe, events и логи. Внесите одну намеренную ошибку и объясните, какой компонент и какая condition показывают её причину. Production-секреты и данные в упражнении не используйте.
