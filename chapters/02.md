# Kubernetes: подробный учебный материал — API-объекты, YAML и kubectl: как декларация меняет кластер

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## Kubernetes API как контракт

kubectl — только один из клиентов API наряду с GitOps-контроллером, оператором и CI. В объекте apiVersion выбирает группу/версию, kind — тип, metadata — идентичность и метаданные, spec — желаемое состояние. Имя можно использовать повторно после удаления, UID относится к конкретному экземпляру. Namespace ограничивает область имён, но не заменяет сетевую изоляцию или RBAC.

## YAML, labels и annotations

YAML чувствителен к отступам и типам: строка, число и boolean не взаимозаменяемы. Labels служат для отбора и группировки; annotations — для неселектируемых метаданных. Не добавляйте в labels request_id или иной почти уникальный идентификатор: это раздувает индексы и кардинальность. Selector — контракт контроллера и Service; ошибка selector часто выглядит как «ресурс создан, но трафика нет».

## Операции kubectl

create создаёт объект и конфликтует с уже существующим; replace отправляет полное представление и может затереть чужие поля; patch меняет часть объекта. apply применяет декларативную конфигурацию. Server-side apply хранит владельца полей (field manager) и обнаруживает конфликт, когда два актора пытаются менять одно поле. Выберите единый способ управления каждым полем, иначе GitOps, оператор и ручной kubectl edit будут перезаписывать друг друга.

## Удаление и finalizers

Удаление устанавливает deletionTimestamp. Finalizer удерживает объект, пока контроллер не завершит внешнюю очистку, например удаление балансировщика или тома. Снять finalizer вручную можно только после проверки внешнего ресурса: иначе API исчезнет, а счёт/данные останутся. ownerReferences связывает владельца и потомка; некорректно созданная связь способна удалить не тот объект через garbage collector.

## Процесс изменения

Храните декларации в Git, проверяйте серверной схемой, смотрите diff, применяйте от контролируемой identity и ожидайте conditions. После изменения проверяйте пользовательский путь, а не только HTTP-ответ apply. Secret не должен лежать в обычном YAML: base64 не является шифрованием; нужны ограниченный RBAC, шифрование at rest и внешний менеджер секретов при необходимости.

## Официальные источники для темы

managedFields полезны для расследования ownership, но увеличивают YAML и обычно не являются частью желаемой конфигурации. В репозитории не храните status, uid, resourceVersion, creationTimestamp и управляемые поля: это наблюдаемое/серверное состояние, а не декларация.

Server-side apply хранит field management: API знает, какой manager владеет конкретным полем. Если GitOps-контроллер и ручной kubectl пытаются менять один путь поля, конфликт должен стать видимым, а не тихо затереть конфигурацию. Force conflicts применяют только с пониманием, какого владельца вы вытесняете и не разрушит ли это оператор.

## Углубление: server-side apply и ownership

generation меняется при изменении spec, а observedGeneration в status обычно сообщает, что controller обработал эту версию. Сравнение этих полей полезнее фиксированного sleep в CI: если observedGeneration отстаёт, контроллер ещё не согласовал новую декларацию или сообщает condition с причиной.

resourceVersion — версия состояния объекта/коллекции, которую API использует для согласованного list/watch. Контроллер сначала получает актуальный список, затем watch-ит изменения и кладёт ключ объекта в очередь reconcile. Reconcile читает текущее состояние заново, сравнивает его с desired state и делает минимальное идемпотентное действие. Повтор события и рестарт controller не должны создавать второй внешний ресурс.

## Углубление: resourceVersion и согласование

После изменения API server не запускает контейнер сам. Он уведомляет наблюдающие контроллеры через list/watch, а каждый controller создаёт следующий объект. Отсюда следует eventual consistency: успешный HTTP-ответ apply ещё не обещает, что ReplicaSet, Pod, endpoint или внешний балансировщик уже готовы.

Клиент обращается к kube-apiserver по TLS, проходит аутентификацию, авторизацию, admission и валидацию схемы. Только затем объект записывается в etcd и ответ возвращается клиенту. 401 означает, что identity не установлена; 403 — identity известна, но verb запрещён; validation/admission error означает, что доступ есть, но объект нарушает правило. Эти причины нельзя лечить одинаково и не нужно проверять их выдачей cluster-admin.

## Домашняя практика

В учебном кластере создайте минимальный пример рассматриваемой сущности. До apply выполните server-side dry-run и diff. После применения сохраните YAML, describe, events и логи.
