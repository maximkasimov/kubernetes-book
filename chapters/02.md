# Kubernetes: API-объекты, YAML и kubectl — как декларация превращается в работающую систему

Эта глава объясняет основной способ работы с Kubernetes. Инженер не «запускает контейнер командой»; он описывает желаемое состояние в API-объекте, отправляет его в Kubernetes API, а контроллеры кластера постепенно приводят реальную систему к этому состоянию. Это важное изменение мышления. Если понимать цепочку целиком, ошибки YAML, пустой Service, зависшее удаление или неожиданный откат перестают выглядеть магией.

## 1. Что такое Kubernetes API и зачем он нужен

Kubernetes API — центральный контракт кластера. Через него создаются Pod, Deployment, Service, Secret, PersistentVolumeClaim и другие объекты. Один и тот же API используют человек с утилитой kubectl, GitOps-контроллер, оператор, CI/CD-система и внутренние компоненты Kubernetes. Поэтому kubectl не управляет контейнерами напрямую: он лишь отправляет HTTP-запрос kube-apiserver.

Представьте простой запрос: инженер применяет описание Deployment. kube-apiserver принимает запрос по TLS, определяет личность клиента, проверяет его права, применяет политики admission, проверяет структуру объекта и только после этого сохраняет объект в etcd. Ответ об успешном применении означает именно «новая декларация записана в API». Он ещё не означает, что Pod создан, образ скачан и приложение принимает трафик.

После записи объекта контроллер Deployment замечает изменение через механизм list/watch. Он создаёт ReplicaSet; ReplicaSet создаёт нужное число Pod; scheduler выбирает для каждого Pod подходящий узел; kubelet на этом узле запускает контейнеры. Service и контроллер EndpointSlice затем замечают готовые Pod и добавляют их адреса в точки назначения. Каждый шаг выполняется отдельно и может занять время или завершиться ошибкой.

Именно поэтому после изменения нужно проверять не только ответ kubectl, но и результат, ради которого оно сделано: condition Deployment, готовность Pod, события, EndpointSlice и, наконец, пользовательский HTTP-запрос.

## 2. Из чего состоит объект Kubernetes

Почти любой объект имеет четыре логические части: apiVersion, kind, metadata и spec. Их полезно воспринимать как ответ на четыре вопроса.

- apiVersion: по какой версии API сервер должен прочитать этот объект. Например, apps/v1 для Deployment.
- kind: какой это тип объекта, например Deployment или Service.
- metadata: как объект называется, где находится и какими служебными метками обладает.
- spec: какое состояние требуется получить.

У созданного объекта сервер также заполняет status. Это не «вторая версия spec». В spec автор описывает намерение: например, три реплики приложения и образ определённой версии. В status контроллер сообщает наблюдаемое состояние: сколько реплик уже создано, доступны ли они, обработал ли контроллер последнюю версию декларации и почему он не смог это сделать.

У каждого экземпляра есть UID. Имя можно удалить и использовать повторно, а UID относится к конкретному существованию объекта. Например, Pod с именем api-7f8c9 после пересоздания может иметь то же имя, но это уже другой объект с другим UID. Для связи с внешним ресурсом или расследования гонок важен UID, а не только name.

Namespace создаёт отдельную область имён: в namespaces team-a и team-b одновременно могут существовать Service с именем api. Но namespace сам по себе не закрывает сетевой доступ, не выдаёт права и не шифрует данные. Для этого отдельно настраивают NetworkPolicy, RBAC и, при необходимости, шифрование данных в etcd.

## 3. YAML — это запись API-объекта, а не программа

YAML удобен человеку, но Kubernetes получает из него структурированные данные. Отступы в YAML определяют вложенность, а типы значений имеют значение. Строка, число и логическое значение не всегда взаимозаменяемы. Например, значение порта в спецификации контейнера — число, а значение переменной окружения — строка. Если написать конфигурацию не в том поле или на неверном уровне вложенности, API обычно отвергнет её при валидации.

Минимальный фрагмент Deployment можно читать так:

    apiVersion: apps/v1
    kind: Deployment
    metadata:
      name: web
      namespace: demo
    spec:
      replicas: 3
      selector:
        matchLabels:
          app: web
      template:
        metadata:
          labels:
            app: web
        spec:
          containers:
          - name: web
            image: registry.example/web:1.4.0

selector говорит Deployment, какие Pod он считает своими. Метка app: web в шаблоне Pod обязана соответствовать selector. Если их изменить несогласованно, Deployment не сможет корректно управлять созданными Pod. Это не формальная деталь: контроллеры и Service строят связи по меткам, а не по человеческим предположениям о названии приложения.

Не добавляйте в Git поля, которые выдаёт API-сервер: uid, resourceVersion, creationTimestamp, managedFields и status. Они описывают конкретный уже существующий объект или наблюдаемое состояние, а не желаемую конфигурацию. Их перенос в манифест часто вызывает конфликты, ложные различия в Git и непредсказуемые обновления.

## 4. Метки и аннотации: похожи внешне, различаются назначением

Метка (label) — короткая пара ключ–значение, по которой Kubernetes выбирает и группирует объекты. Service выбирает Pod по selector из labels. Deployment выбирает свои Pod тем же способом. Команды kubectl могут выбрать все объекты команды или окружения по метке. Хорошие labels имеют небольшой, контролируемый набор значений: app=payments, environment=prod, component=api.

Аннотация (annotation) — тоже метаданные, но не средство отбора. В ней размещают информацию для инструмента или контроллера: ссылку на runbook, имя владельца, параметры конкретного ingress-контроллера, checksum конфигурации. Аннотации не нужно использовать как замену labels, если объект требуется выбрать selector-ом.

Особенно опасны почти уникальные labels, например request_id, user_id или случайный идентификатор сборки для каждого Pod. Они увеличивают число комбинаций меток и объём индексов во многих системах наблюдаемости и управления. Метка должна описывать устойчивый класс объекта, а конкретный запрос или пользователя лучше искать в журнале событий или трассировке.

## 5. Что делают create, apply, replace и patch

kubectl create создаёт новый объект. Если такой объект уже есть, команда завершится конфликтом. Это удобно для одноразового создания, но плохо подходит для постоянной декларации, которую требуется безопасно менять.

kubectl apply сравнивает описанную конфигурацию с объектом в API и применяет нужные изменения. Это обычный выбор для управляемых манифестов. В production лучше явно определить владельца конфигурации: GitOps-контроллер или CI/CD. Ручная правка работающего объекта должна быть исключением, иначе следующий синхронизирующий запуск вернёт состояние из Git и «исчезнувшая» ручная правка будет совершенно ожидаемым результатом.

kubectl replace отправляет полное новое представление объекта. Если шаблон устарел или из него исчезли поля, он способен затереть чужие изменения. kubectl patch изменяет часть объекта и полезен для контролируемой точечной операции, но не заменяет хранение целой декларации в Git.

Server-side apply — серверный вариант декларативного применения. API хранит сведения о владельцах отдельных полей. Если GitOps-контроллер управляет образом контейнера, а другой клиент пытается поменять тот же путь поля, сервер возвращает конфликт вместо тихой перезаписи. Это защищает не весь объект целиком, а конкретные поля. Параметр force-conflicts нельзя считать обычным способом «починить деплой»: он отнимает владение у другого клиента и требует понимания последствий.

## 6. Почему после apply система меняется не мгновенно

Kubernetes использует согласование состояния (reconciliation). Контроллер читает текущее состояние, сравнивает его с требуемым состоянием и выполняет минимальное действие. Если действие не удалось, он повторит попытку. Поэтому контроллеры должны быть идемпотентными: повторная обработка одного события не должна создавать второй одинаковый внешний балансировщик или второй том.

Поле metadata.generation увеличивается при изменении spec. Многие контроллеры записывают в status.observedGeneration номер декларации, которую они уже обработали. Когда generation больше observedGeneration, контроллер ещё не подтвердил новую версию. Это надёжнее фиксированной паузы в CI/CD: вместо ожидания «30 секунд на всякий случай» проверяют состояние именно того изменения, которое было отправлено.

В status.conditions обычно находятся человечески полезные причины: Progressing, Available, Ready и похожие статусы зависят от типа объекта. При ошибке образа, нехватке ресурсов, неверной конфигурации или отказе внешнего провайдера важно читать condition и events, а не немедленно повторять apply. Повторный запрос не исправит отсутствующий Secret или невозможность scheduler разместить Pod.

## 7. Удаление, finalizer и владельцы объектов

Когда выполняется удаление, Kubernetes часто не стирает объект немедленно. Он записывает deletionTimestamp. Если у объекта есть finalizer, объект остаётся видимым, пока контроллер не завершит внешнюю очистку: например, не удалит облачный балансировщик, диск или запись DNS. Это предохраняет от ситуации, где запись в API исчезла, а платный внешний ресурс остался бесхозным.

Удалять finalizer вручную допустимо только после проверки внешней системы. Иначе можно скрыть проблему, а не решить её. В расследовании сначала определяют, какой контроллер установил finalizer, почему его reconcile не завершился, существует ли внешний ресурс и безопасно ли его удалить или отвязать.

ownerReferences связывают объект-владельца и зависимый объект. Например, ReplicaSet владеет Pod, а Deployment — ReplicaSet. Сборщик мусора Kubernetes удаляет зависимые объекты согласно этим связям. Неверная ownerReference опасна: она способна удалить ресурс при исчезновении не того владельца. Не создавайте такие связи вручную без точного понимания жизненного цикла обоих объектов.

## 8. Безопасный процесс изменения

Надёжное изменение начинается с версии манифеста в Git и понятного владельца каждого поля. Перед применением полезны серверная проверка без записи и просмотр различий:

    kubectl apply --server-side --dry-run=server -f deployment.yaml
    kubectl diff -f deployment.yaml

dry-run=server проходит серверную валидацию и admission, но не сохраняет объект. diff показывает ожидаемое изменение относительно кластера. Эти команды не заменяют тестовое окружение и ревью, но ловят многие ошибки до production.

После применения проверяют последовательность: состояние Deployment, созданные Pod, события, готовые EndpointSlice, доступность Service и пользовательский сценарий. Если в манифесте есть Secret, base64-кодирование не считается шифрованием. Секреты не размещают в открытом Git; применяют минимальные RBAC-права, шифрование данных в etcd и при необходимости внешний менеджер секретов.

## Домашняя практика

В учебном кластере создайте Deployment с одной репликой и Service, который выбирает этот Pod по label. Выполните server-side dry-run и diff, примените манифест и проследите путь объекта: Deployment, ReplicaSet, Pod, EndpointSlice. Затем намеренно измените selector Service так, чтобы он не совпадал с label Pod, и объясните, почему Pod остаётся Ready, но Service больше не имеет адресов. Production-секреты и реальные ключи в упражнении не используйте.
