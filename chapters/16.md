# Kubernetes: подробный учебный материал — Gateway API: современная модель ролей, классов и маршрутов

## Почему Gateway API

Gateway API развивает модель Ingress: разделяет владельца инфраструктуры и владельца маршрута, стандартизирует HTTP, TLS, TCP, UDP и gRPC маршрутизацию, а также статус программирования. Вместо vendor annotations используются типизированные ресурсы. Поддержка зависит от конкретного Gateway controller и его GatewayClass.

## GatewayClass и Gateway

GatewayClass описывает класс реализации, управляемый инфраструктурной командой. Gateway — конкретная точка входа с listeners: protocol, port, hostname, TLS и policy допуска маршрутов. Controller создаёт/настраивает реальный dataplane. Gateway object не гарантирует готовый IP: читайте status conditions и addresses, которые сообщил controller.

## Routes и attachment

HTTPRoute, GRPCRoute, TCPRoute, TLSRoute и UDPRoute описывают правила направления. Route привязывается через parentRefs к Gateway/listener. Listener allowedRoutes контролирует, какие namespaces/kinds могут прикрепляться. Это создаёт понятную делегацию: platform владеет портами и сертификатами, приложение — только разрешённым маршрутом.

## Статус как контракт

Gateway API активно использует conditions Accepted, Programmed, ResolvedRefs и другие статусы. Accepted означает, что объект принят реализацией; Programmed — что dataplane настроен, но точный смысл сверяют с документацией controller. Проверка YAML без этих conditions не доказывает работоспособность маршрута.

## Переход от Ingress

Не мигрируйте механически аннотации. Сначала опишите реальный listener, host/path, TLS, source IP, timeouts и тип backend. Разделите стандартное поле Gateway API и vendor policy. Запустите параллельный путь, сравните user-visible поведение, затем переключайте DNS/трафик контролируемо.

Gateway API introduction: https://gateway-api.sigs.k8s.io/ . Gateway API concepts: https://gateway-api.sigs.k8s.io/concepts/api-overview/ . Kubernetes Gateway API: https://kubernetes.io/docs/concepts/services-networking/gateway/.

## Официальные источники для темы

Проверяйте Gateway status addresses и listener conditions, затем Route status для каждого parentRef. Accepted указывает, что конфигурация принята, ResolvedRefs — что ссылки разрешимы, Programmed — что dataplane настроен по смыслу реализации. Условие не заменяет проверку реального запроса, но исключает догадки о том, где контроллер отверг объект.

## Углубление: status

parentRefs — запрос Route на привязку, но окончательное решение принимает listener/implementation. allowedRoutes и ReferenceGrant — разные механизмы: первый регулирует, кто может прикрепить Route к listener; второй — кто может сослаться на ресурс в другом namespace.

GatewayClass выбирает реализацию и обычно принадлежит platform team. Gateway описывает конкретный dataplane edge и listeners, включая допустимые namespaces/routes. Route принадлежит приложению и описывает backend. Такая декомпозиция не только «красивее Ingress»: она не позволяет приложению произвольно открыть listener, использовать общий сертификат или сослаться на чужой backend без разрешения.
