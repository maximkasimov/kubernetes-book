# Kubernetes: подробный учебный материал — Конфигурация, задания и секреты: границы между кодом, данными и доступом

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## ConfigMap и Secret

ConfigMap хранит неконфиденциальную конфигурацию, Secret — чувствительные данные. В стандартном Kubernetes Secret base64-кодирован, а не зашифрован. Право get/list/watch на Secrets критично: субъект с ним читает значения. Шифрование at rest и аудит доступа настраиваются отдельно.

Изучайте ресурс через Kubernetes API, а не как отдельный YAML-файл. API server хранит желаемое состояние, а контроллеры асинхронно приводят кластер к нему. Поэтому момент успешного apply и момент готовности приложения различны.

Для любого объекта различайте spec и status. spec задаёт намерение, status и conditions сообщают результат работы контроллеров. Conditions с полями type, status, reason и message обычно точнее краткой колонки kubectl get.

События (Events) полезны как временная диагностическая подсказка. Их нельзя считать долговременным аудитом: сохраняйте важные факты в системе логирования и runbook. Всегда проверяйте ownerReferences, чтобы понять, какой контроллер создал объект.

Безопасная диагностика идёт сверху вниз: пользовательский результат, ресурс верхнего уровня, его потомки, node/runtime и внешняя зависимость. Массовое удаление Pod или выдача cluster-admin может скрыть причину, а не устранить её.

## Доставка конфигурации

Конфигурация может попасть через environment, args или смонтированный файл. Environment уже запущенного контейнера не изменяется. Смонтированный ConfigMap/Secret может обновиться с задержкой, но приложение должно перечитать файл; subPath обновлений не получает. Для предсказуемости используют версионированное имя либо checksum в Pod template и контролируемый rollout.

## Job

Job определяет completions, parallelism, backoffLimit и activeDeadlineSeconds. Успешный exit code означает успех процесса, не бизнес-транзакции. При повторе «создать пользователя» или «списать платёж» должна применяться идемпотентная операция на целевой стороне. Результат сохраняйте устойчиво, не только в логах короткоживущего Pod.

## Внешние секреты и ротация

Vault, облачный secret manager или иной внешний источник сокращает распространение постоянных ключей, но добавляет зависимости: identity, сеть, ротация и поведение при недоступности. Ротация закончена только когда новый credential создан, потребитель его применил, соединения проверены и старое значение безопасно удалено.

## Углубление: service account token

Новые и старые конфигурации должны быть совместимы во время rollout. Например, сначала приложение принимает оба ключа, затем меняется config, после проверки удаляется старый ключ. Обратный порядок создаёт окно, когда часть Pods не может стартовать.

Когда ConfigMap/Secret смонтирован томом, kubelet периодически обновляет projection, но приложение может держать файл открытым или читать значение только при старте. Поэтому «изменили ConfigMap» не равно «изменили конфигурацию всех реплик». Выберите один явный контракт: restart по checksum, reload по signal/endpoint или версионированный ресурс.

## Углубление: обновление конфигурации

ServiceAccount, используемый приложением, как правило, не должен иметь права читать Secret API: kubelet уже монтирует конкретно объявленный Secret. Это уменьшает последствия компрометации приложения. Отдельно проверьте, не попадает ли значение в аргументы процесса, debug output, backup или CI logs.

Secret проходит через API, etcd, kubelet и volume/environment контейнера. Encryption at rest защищает данные в etcd, но не заменяет TLS, RBAC и защиту node: kubelet должен получить Secret для запуска Pod. Ограничивайте list/watch Secrets особенно строго: list часто раскрывает больше, чем точечный get.

## Домашняя практика

В учебном кластере создайте минимальный пример рассматриваемой сущности. До apply выполните server-side dry-run и diff. После применения сохраните YAML, describe, events и логи. Внесите одну намеренную ошибку и объясните, какой компонент и какая condition показывают её причину. Production-секреты и данные в упражнении не используйте.
