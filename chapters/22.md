# Kubernetes: подробный учебный материал — Диагностика Kubernetes: системный путь от симптома к причине

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## Факт и граница

Начните с точного симптома, времени, области и пользовательского результата. «Сервис недоступен» разделите на DNS, TLS, Gateway, Service, endpoints, Pod, dependency и node. Зафиксируйте generation/observedGeneration, conditions, events, node, image digest, restart count и последние изменения до любых действий.

Изучайте ресурс через Kubernetes API, а не как отдельный YAML-файл. API server хранит желаемое состояние, а контроллеры асинхронно приводят кластер к нему. Поэтому момент успешного apply и момент готовности приложения различны.

Для любого объекта различайте spec и status. spec задаёт намерение, status и conditions сообщают результат работы контроллеров. Conditions с полями type, status, reason и message обычно точнее краткой колонки kubectl get.

События (Events) полезны как временная диагностическая подсказка. Их нельзя считать долговременным аудитом: сохраняйте важные факты в системе логирования и runbook. Всегда проверяйте ownerReferences, чтобы понять, какой контроллер создал объект.

Безопасная диагностика идёт сверху вниз: пользовательский результат, ресурс верхнего уровня, его потомки, node/runtime и внешняя зависимость. Массовое удаление Pod или выдача cluster-admin может скрыть причину, а не устранить её.

## Workload

Deployment → ReplicaSet → Pod — базовая цепочка. Pending означает проверить scheduler, resources, taints, affinity, quota, PVC. ContainerCreating ведёт к image, mount, runtime, CNI. CrashLoopBackOff ведёт к previous logs, exit code, command, config/secret и probes. Running not Ready ведёт к readiness и приложению.

## Сеть

Для сетевого сбоя определите source Pod/namespace, destination IP/name/port, DNS и first failing hop. Проверяйте policy, EndpointSlice, Service port/targetPort, CNI agent и routes. Не заменяйте тест из Pod тестом с node. Для ingress добавьте listener/Gateway/Route conditions, certificate и внешний балансировщик.

## Control plane

При API errors отделите authentication, authorization, admission, validation и etcd/control-plane availability. 401, 403 и admission denial имеют разные причины. Проверяйте API latency/errors, controller/scheduler health, etcd quorum, webhooks и сертификаты. Уже работающие Pods могут отвечать при недоступном API — это не означает здоровый кластер.

## Официальные источники для темы

Эскалируйте по границе владения: app team — process/config, platform — scheduler/RBAC/cluster API, network — route/BGP/firewall, storage — backend/CSI. Перед передачей соберите source/destination, namespace, node, time, error, подтверждённые/исключённые слои. Это быстрее, чем сообщение «Kubernetes не работает».

## Углубление: эскалация

Снимайте status YAML, events, логи, метрики/flows и recent deployment/Git history до cleanup. Это превращает postmortem из мнения в проверяемую цепочку. После восстановления добавляйте guardrail: тест, admission policy, alert или runbook, который ловит тот же класс ошибки раньше.

До действия сформулируйте гипотезу, ожидаемый наблюдаемый эффект и rollback. Меняйте одну переменную, фиксируйте timestamp и повторяйте исходный тест. Массовый rollout restart, удаление всех Pods и отключение policy часто делают систему «зелёной», но стирают доказательства и могут усиливать outage.

## Углубление: изменение как эксперимент

При Pending фиксируйте exact FailedScheduling message: insufficient CPU, taint, affinity, PVC topology, quota и max pods имеют разные владельцы. При ContainerCreating не перепрыгивайте к приложению: inspect events MountVolume/FailedCreatePodSandBox/ErrImagePull и логи соответствующего агента.

Одно внешнее 502 не указывает на backend. Сначала определите, какой edge вернул ответ, затем listener/route status, upstream Service/EndpointSlice, readiness Pod, NetworkPolicy и логи приложения. Если connection timeout, разделите DNS, TCP handshake, TLS и HTTP: каждый протоколный этап сужает владельца дефекта.

## Домашняя практика

В учебном кластере создайте минимальный пример рассматриваемой сущности. До apply выполните server-side dry-run и diff. После применения сохраните YAML, describe, events и логи. Внесите одну намеренную ошибку и объясните, какой компонент и какая condition показывают её причину. Production-секреты и данные в упражнении не используйте.
