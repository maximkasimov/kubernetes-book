# Kubernetes: подробный учебный материал — Диагностика Kubernetes: системный путь от симптома к причине

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## Факт и граница

Начните с точного симптома, времени, области и пользовательского результата. «Сервис недоступен» разделите на DNS, TLS, Gateway, Service, endpoints, Pod, dependency и node. Зафиксируйте generation/observedGeneration, conditions, events, node, image digest, restart count и последние изменения до любых действий.

## Workload

Deployment → ReplicaSet → Pod — базовая цепочка. Pending означает проверить scheduler, resources, taints, affinity, quota, PVC. ContainerCreating ведёт к image, mount, runtime, CNI. CrashLoopBackOff ведёт к previous logs, exit code, command, config/secret и probes. Running not Ready ведёт к readiness и приложению.

## Сеть

Для сетевого сбоя определите source Pod/namespace, destination IP/name/port, DNS и first failing hop. Проверяйте policy, EndpointSlice, Service port/targetPort, CNI agent и routes. Не заменяйте тест из Pod тестом с node. Для ingress добавьте listener/Gateway/Route conditions, certificate и внешний балансировщик.

## Control plane

При API errors отделите authentication, authorization, admission, validation и etcd/control-plane availability. 401, 403 и admission denial имеют разные причины. Проверяйте API latency/errors, controller/scheduler health, etcd quorum, webhooks и сертификаты. Уже работающие Pods могут отвечать при недоступном API — это не означает здоровый кластер.

## Завершение инцидента

После изменения повторите исходный пользовательский тест, затем проверьте метрики/alerts. Сохраните причину, доказательства, действие, rollback и профилактику. Если дефект в selector/policy/admission/quota, добавьте проверку в CI или admission — иначе следующий apply вернёт проблему.

Application troubleshooting: https://kubernetes.io/docs/tasks/debug/debug-application/ . Cluster troubleshooting: https://kubernetes.io/docs/tasks/debug/debug-cluster/ . Debug Services: https://kubernetes.io/docs/tasks/debug/debug-application/debug-service/.

## Официальные источники для темы

Эскалируйте по границе владения: app team — process/config, platform — scheduler/RBAC/cluster API, network — route/BGP/firewall, storage — backend/CSI. Перед передачей соберите source/destination, namespace, node, time, error, подтверждённые/исключённые слои. Это быстрее, чем сообщение «Kubernetes не работает».

## Углубление: эскалация

Снимайте status YAML, events, логи, метрики/flows и recent deployment/Git history до cleanup. Это превращает postmortem из мнения в проверяемую цепочку. После восстановления добавляйте guardrail: тест, admission policy, alert или runbook, который ловит тот же класс ошибки раньше.

До действия сформулируйте гипотезу, ожидаемый наблюдаемый эффект и rollback. Меняйте одну переменную, фиксируйте timestamp и повторяйте исходный тест. Массовый rollout restart, удаление всех Pods и отключение policy часто делают систему «зелёной», но стирают доказательства и могут усиливать outage.

## Углубление: изменение как эксперимент

При Pending фиксируйте exact FailedScheduling message: insufficient CPU, taint, affinity, PVC topology, quota и max pods имеют разные владельцы. При ContainerCreating не перепрыгивайте к приложению: inspect events MountVolume/FailedCreatePodSandBox/ErrImagePull и логи соответствующего агента.

Одно внешнее 502 не указывает на backend. Сначала определите, какой edge вернул ответ, затем listener/route status, upstream Service/EndpointSlice, readiness Pod, NetworkPolicy и логи приложения. Если connection timeout, разделите DNS, TCP handshake, TLS и HTTP: каждый протоколный этап сужает владельца дефекта.

## Практический runbook: от симптома к доказательству

Перед началом задайте контекст, чтобы команды не попали в другой кластер или namespace. Команда `kubectl config current-context` должна показывать ожидаемый контур; `kubectl get ns` — ожидаемые пространства имён. Для инцидента сохраните время в UTC, namespace и имя workload: это позволит сопоставить события, логи и метрики.

```bash
kubectl config current-context
kubectl -n payments get deploy payments -o wide
kubectl -n payments get deploy payments -o yaml
kubectl -n payments get events --sort-by=.metadata.creationTimestamp
```

Не копируйте в тикет полный YAML Secret или вывод команд, содержащий токены. Для объекта верхнего уровня сначала прочитайте `status.conditions`, `generation` и `observedGeneration`. Если `observedGeneration` отстаёт, контроллер ещё не обработал последнюю декларацию; рестарт Pod в этот момент не является диагностикой причины.

## Сценарий: Deployment не получает готовые реплики

Начинайте с цепочки владельцев: Deployment создаёт ReplicaSet, ReplicaSet создаёт Pod. Это позволяет не спутать дефект rollout с дефектом конкретного контейнера. В примере ниже `kubectl rollout status` ждёт condition Deployment, а не просто появления Pod в списке.

```bash
kubectl -n payments rollout status deployment/payments --timeout=90s
kubectl -n payments describe deployment payments
kubectl -n payments get rs -l app=payments -o wide
kubectl -n payments get pods -l app=payments -o wide
kubectl -n payments describe pod <pod-name>
```

`Pending` с `FailedScheduling` означает, что контейнер ещё не запускался. В сообщении scheduler ищите конкретную причину: `Insufficient cpu`, `Insufficient memory`, taint без toleration, node affinity, quota, topology или PVC. Не увеличивайте requests вслепую: сначала сравните их с `allocatable` и фактическим потреблением.

```bash
kubectl -n payments get pod <pod-name> -o jsonpath='{.status.conditions[?(@.type=="PodScheduled")].message}{"\n"}'
kubectl describe node <node-name>
kubectl -n payments describe resourcequota
kubectl -n payments describe limitrange
kubectl -n payments get pvc
```

Если Pod уже запускается, но перезапускается, соберите текущий и предыдущий stdout/stderr, код завершения и события. `--previous` особенно важен при CrashLoopBackOff: текущий контейнер часто ещё не успел написать полезный лог.

```bash
kubectl -n payments logs <pod-name> -c app --timestamps
kubectl -n payments logs <pod-name> -c app --previous --timestamps
kubectl -n payments get pod <pod-name> -o jsonpath='{range .status.containerStatuses[*]}{.name}{" exit="}{.state.terminated.exitCode}{" reason="}{.state.terminated.reason}{"\n"}{end}'
kubectl -n payments describe pod <pod-name>
```

`Running` вместе с `Ready=False` — другая ветка. Проверьте readiness probe, endpoint приложения, конфигурацию и зависимости. Не меняйте liveness probe как первое действие: она отвечает за рестарт и может превратить временную задержку в цикл рестартов.

## Сценарий: Service есть, но соединение не проходит

Проверка должна идти из Pod того же namespace, а не с node администратора. Сначала сравните selector Service с labels Pod, затем EndpointSlice и порты. Наличие DNS-имени или ClusterIP само по себе не означает, что есть ready backend.

```bash
kubectl -n payments get svc payments -o yaml
kubectl -n payments get pods -l app=payments --show-labels
kubectl -n payments get endpointslice -l kubernetes.io/service-name=payments -o yaml
kubectl -n payments get pod <pod-name> -o jsonpath='{.status.conditions[?(@.type=="Ready")].status}{"\n"}'
```

Для контролируемого сетевого теста используйте временный диагностический Pod только в разрешённом test/production namespace и удаляйте его после проверки. Образ и право создания Pod должны быть заранее согласованы с политикой платформы.

```bash
kubectl -n payments run net-debug --rm -it --restart=Never \
  --image=nicolaka/netshoot -- sh
# внутри Pod:
nslookup payments
nc -vz payments 8080
curl -sv http://payments:8080/health
```

Если Pod IP доступен, а Service нет — исследуйте selector, EndpointSlice, targetPort и Service dataplane. Если Service работает, но только один namespace получает timeout, проверьте NetworkPolicy в namespace источника и назначения. Если DNS не отвечает, отдельно проверяйте CoreDNS Service/endpoints и правила UDP/TCP 53; не подменяйте DNS адресом Pod.

## Сценарий: Gateway или TLS-маршрут возвращает 404, 502 либо timeout

Для Gateway API сначала проверяют не логи backend, а conditions Gateway и Route. `Accepted=False`, `ResolvedRefs=False` и `Programmed=False` отвечают на разные вопросы: принят ли объект, разрешены ли ссылки и настроен ли dataplane. Сохраните имя listener, hostname, путь, SNI и точный код ответа.

```bash
kubectl -n edge get gateway public -o yaml
kubectl -n payments get httproute payments -o yaml
kubectl -n payments describe httproute payments
kubectl -n payments get referencegrant
kubectl -n payments get svc payments
kubectl -n payments get endpointslice -l kubernetes.io/service-name=payments
```

Проверка TLS должна обращаться к фактическому hostname, иначе SNI и маршрут могут не совпасть. Не передавайте приватный ключ или содержимое Secret в терминал/тикет.

```bash
curl -sv --resolve app.example.internal:443:<gateway-ip> https://app.example.internal/health
openssl s_client -connect <gateway-ip>:443 -servername app.example.internal </dev/null
```

## Сценарий: API control plane недоступен или отвечает ошибками

Сначала отличите локальную проблему kubeconfig/DNS от состояния кластера. Проверка `/readyz` полезна только для разрешённого административного клиента и не заменяет мониторинг etcd/API. При потере API уже работающие Pod могут продолжать отвечать, но новые rollout, scheduler и контроллеры деградируют.

```bash
kubectl get --raw='/livez?verbose'
kubectl get --raw='/readyz?verbose'
kubectl get nodes
kubectl get --raw='/version'
kubectl auth can-i get pods -n payments --as=system:serviceaccount:payments:app
```

401, 403 и webhook/admission error расследуются раздельно. Для admission собирают полный ответ API, состояние service/endpoints webhook, сертификат и NetworkPolicy. Не меняйте `failurePolicy` на `Ignore` и не удаляйте webhook как первую реакцию: это изменение безопасности всего кластера. Операции с etcd quorum, восстановлением или удалением участников выполняются только по проверенному HA/restore runbook.

## Домашняя практика

В учебном кластере создайте минимальный пример рассматриваемой сущности. До apply выполните server-side dry-run и diff. После применения сохраните YAML, describe, events и логи. Внесите одну намеренную ошибку и объясните, какой компонент и какая condition показывают её причину. Production-секреты и данные в упражнении не используйте.
