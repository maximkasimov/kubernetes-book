# Kubernetes: подробный учебный материал — RBAC, admission и безопасность Kubernetes

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## Identity и ServiceAccount

Kubernetes API сначала устанавливает identity, затем authorizer решает разрешён ли verb над resource/subresource в namespace или кластере. ServiceAccount — identity Pod; automount токена не должен быть включён по привычке. Не путайте доступ приложения к API с доступом человека через kubeconfig: они требуют разных ролей, audit и сроков жизни credential.

Изучайте ресурс через Kubernetes API, а не как отдельный YAML-файл. API server хранит желаемое состояние, а контроллеры асинхронно приводят кластер к нему. Поэтому момент успешного apply и момент готовности приложения различны.

Для любого объекта различайте spec и status. spec задаёт намерение, status и conditions сообщают результат работы контроллеров. Conditions с полями type, status, reason и message обычно точнее краткой колонки kubectl get.

События (Events) полезны как временная диагностическая подсказка. Их нельзя считать долговременным аудитом: сохраняйте важные факты в системе логирования и runbook. Всегда проверяйте ownerReferences, чтобы понять, какой контроллер создал объект.

Безопасная диагностика идёт сверху вниз: пользовательский результат, ресурс верхнего уровня, его потомки, node/runtime и внешняя зависимость. Массовое удаление Pod или выдача cluster-admin может скрыть причину, а не устранить её.

## RBAC

Role действует в namespace, ClusterRole — кластерно или может быть привязан в namespace. RoleBinding назначает Role/ClusterRole субъектам в namespace, ClusterRoleBinding — по всему кластеру. Давайте минимальные verbs и resources, включая subresources pods/log, pods/exec, secrets. Проверяйте доступ от имени реального ServiceAccount, а не cluster-admin.

## Admission и политики

Mutating admission добавляет/изменяет объект, validating admission принимает/отклоняет его. Webhook — критичная зависимость API: TLS, service endpoints, timeout, failurePolicy и selector могут остановить развёртывания либо ослабить защиту. Политика должна быть точной: запрещать privileged, требовать resources/labels, ограничивать registry, но не маскировать непонятные defaults.

## Pod security

Контейнеры запускайте non-root, без privileged, с drop capabilities, read-only root filesystem там, где возможно, seccomp и минимальными volumes. Pod Security Standards/Admission задают базовые профили, но не заменяют проверку образов, NetworkPolicy и RBAC. Исключения оформляйте явно по namespace/политике и регулярно пересматривайте.

## Supply chain и audit

Используйте фиксированные image digests или контролируемые теги, сканирование, подпись и registry policy. Secrets не должны попадать в образ и логи. Audit API помогает понять, кто изменил объект; настройка аудита требует баланса между полнотой и объёмом/конфиденциальностью. Безопасность — набор проверяемых границ, а не один включённый флаг.

## Официальные источники для темы

Привилегированный Pod, hostNetwork, hostPID, hostPath и доступ к socket runtime размывают границу контейнера с node. Такие разрешения допустимы для тщательно проверенных системных компонентов, но не обычных приложений. Admission должен ограничивать их, а namespace системных агентов иметь отдельный ownership и audit.

## Углубление: угрозы node

failurePolicy Fail усиливает безопасность, но превращает недоступный webhook в блокировку API для совпавших объектов; Ignore уменьшает доступность риска, но может пропустить запрет. Выбор фиксируется для каждого policy отдельно, с runbook и проверенным планом ремонта, а не меняется в панике.

Mutating admission может добавлять labels, sidecar, defaults или projected volume; validating admission смотрит финальную, уже изменённую форму. Порядок и взаимодействие webhook критичны: один webhook может изменить объект так, что другой его отклонит. Webhook должен быть HA, иметь валидный CA/certificate, ограниченный scope и измеримые latency/error.

## Углубление: цепочка admission

Не выдавайте wildcard verbs/resources для удобства controller. Сначала определите минимальный API-набор по логам denied запросов и документации компонента. Отдельно ограничивайте создание RoleBinding и изменение admission/policy: это пути косвенной эскалации даже без прямого доступа к Secrets.

RBAC складывается из bindings: субъект может получить разрешение несколькими RoleBinding/ClusterRoleBinding. Проверяйте конкретный verb, resource, subresource и namespace от имени реального identity. Право get pods не даёт pods/log или pods/exec; право читать ConfigMap не даёт читать Secret. Используйте kubectl auth can-i как проверку, но храните policy и тесты доступа декларативно.

## Домашняя практика

В учебном кластере создайте минимальный пример рассматриваемой сущности. До apply выполните server-side dry-run и diff. После применения сохраните YAML, describe, events и логи. Внесите одну намеренную ошибку и объясните, какой компонент и какая condition показывают её причину. Production-секреты и данные в упражнении не используйте.
