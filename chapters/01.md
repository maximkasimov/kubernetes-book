# Kubernetes: архитектура и модель управления

Том 1. Полная теоретическая лекция

## 1. Что Kubernetes решает, а что оставляет вам

Kubernetes — это система управления желаемым состоянием распределённых контейнерных нагрузок. Вы описываете, какие объекты должны существовать: сколько реплик приложения, какие порты доступны, где хранить данные, какие ресурсы требуются. Контроллеры непрерывно сравнивают это описание с наблюдаемым состоянием и пытаются уменьшить расхождение. Kubernetes не является ни просто планировщиком контейнеров, ни готовой платформой «с кнопкой production»: он задаёт API и механизмы, а сеть, хранилище, политика доступа, наблюдаемость и процесс изменений должны быть спроектированы для конкретного кластера.

Главная мысль для инженера: объект YAML — не команда, которую Kubernetes исполняет один раз. Это декларация. Если вручную удалить Pod, контроллер Deployment создаст новый, потому что desired state говорит о требуемом числе реплик. Если отредактировать созданный контроллером Pod напрямую, следующее согласование может отменить изменение. Поэтому production-изменения делают в источнике декларации — Git, Helm values, Kustomize или другом контролируемом механизме, а не в случайной ручной правке потомка.

Kubernetes не гарантирует, что приложение станет отказоустойчивым только от того, что у него несколько Pod. Реплики могут оказаться на одном node, использовать один PVC, зависеть от одного внешнего API или быть недоступны из-за одной NetworkPolicy. Сначала формулируют свойства: сколько реплик/зон переживает сервис, какой RTO и RPO, как обнаруживается отказ. Затем выбирают Deployment, StatefulSet, PDB, affinity, storage и сетевую topology.

## 2. Control plane: единственный источник истины API

Control plane принимает декларации, хранит состояние и принимает глобальные решения. kube-apiserver — HTTP API и точка аутентификации, авторизации, admission и чтения/записи объектов. Клиент kubectl, контроллеры, scheduler и операторы не меняют etcd напрямую: все работают через API server. Это упрощает аудит и согласованность, но делает доступность API и корректность его сертификатов критичными для управления кластером.

etcd хранит состояние Kubernetes как распределённое key-value-хранилище. Потеря или повреждение etcd — не обычная проблема одного Pod: это риск потери самой картины desired state. Для production нужны резервные копии etcd, проверенная процедура восстановления, безопасное обновление и понимание quorum. Запуск трёх экземпляров не даёт магической безопасности, если они находятся в одном домене отказа или команда никогда не проверяла restore на изолированном контуре.

kube-scheduler выбирает node для Pod, опираясь на requests, taints/tolerations, affinity, topology constraints и другие ограничения. kube-controller-manager запускает набор контроллеров, поддерживающих объекты: Deployment, ReplicaSet, Node lifecycle и другие. Cloud-controller-manager, если применим, связывает кластер с облачным API. Разделение ролей важно при диагностике: Pending Pod может быть следствием scheduler/ресурсов, а не проблемой container runtime; объект без endpoints — проблемой readiness или Service selector, а не API server.

## 3. Node: где декларация становится работающим процессом

На каждом node kubelet наблюдает API server и приводит локальное состояние к PodSpec: получает образ через container runtime, создаёт sandbox Pod, вызывает CNI, монтирует volumes и следит за контейнерами. Kubelet не является универсальным менеджером процессов хоста: он отвечает за Pod, созданные через Kubernetes. Если контейнер запущен вручную, Kubernetes не включит его в desired state и не применит к нему readiness, Service или policy.

Container runtime реализует CRI и запускает контейнеры; в типичных кластерах используют containerd или CRI-O. Runtime создаёт контейнеры, а сетевое подключение передаётся CNI-плагину. Это разделение помогает правильно искать проблему: ImagePullBackOff относится к доступу к registry/secret/runtime; ContainerCreating может указывать на volume или CNI; CrashLoopBackOff — на процесс приложения, его конфигурацию или probes. Один статус Pod — это начало расследования, а не диагноз.

kube-proxy традиционно реализует часть поведения Service на node через правила dataplane. Некоторые современные CNI, например Cilium, могут предоставлять эквивалентную обработку Services сами, тогда kube-proxy может быть заменён в поддерживаемой конфигурации. Не делайте вывод по названию продукта: в конкретном кластере нужно проверить, какой dataplane реально отвечает за ClusterIP, NodePort и балансировку, прежде чем менять iptables, eBPF или правила firewall.

## 4. Путь одного manifest от kubectl к Pod

Когда вы выполняете kubectl apply, клиент отправляет manifest в kube-apiserver. API server аутентифицирует пользователя, проверяет RBAC, запускает admission-цепочку, валидирует схему и записывает принятый объект в etcd. С этого момента объект доступен через watch для контроллеров. Server-side apply хранит информацию о владельцах полей, поэтому конфликт двух систем доставки должен быть виден как конфликт управления полем, а не как тихое перетирание YAML.

Если manifest описывает Deployment, Deployment controller создаёт или обновляет ReplicaSet; ReplicaSet controller создаёт Pod. Scheduler видит Pod без nodeName и назначает node. Kubelet выбранного node получает PodSpec и начинает локальную подготовку. Каждый этап публикует status и события. Поэтому диагностика проходит по цепочке: manifest/metadata, status Deployment, ReplicaSet, Pod, scheduling events, kubelet/runtime/CNI, затем readiness и Service endpoints. Перезапускать всё сразу означает потерять полезные признаки этой цепочки.

Успех apply не доказывает готовность сервиса. API мог принять desired state, а Pod может остаться Pending, не скачать образ, не получить IP, не пройти readiness или не попасть в EndpointSlice. Production-приёмка требует сквозного факта: Pod Ready, Service имеет ожидаемый endpoint, разрешённый клиент может выполнить соединение, а наблюдаемость видит метрики/логи. В сетевых последующих томах этот путь будет расширен до CNI, DNS, NetworkPolicy, Ingress и Gateway API.

## 5. Namespace, labels и selectors как контракт платформы

Namespace — логическая область имён и удобная граница для RBAC, quotas, NetworkPolicy и владения. Он не является сетевой изоляцией сам по себе: без NetworkPolicy Pod из другого namespace может иметь сетевой доступ. Аналогично namespace не равен tenant приложения автоматически. В production нужно явно описывать, что namespace изолирует, кто может создавать объекты, какие ресурсы доступны и какие внешние зависимости разрешены.

Labels — основной механизм группировки и выбора объектов. Deployment выбирает Pods, Service выбирает endpoints, NetworkPolicy выбирает Pod и namespaces, мониторинг/CI выбирают объекты по тем же меткам. Ошибка label selector может дать крайне разные симптомы: Service без endpoints, policy не применена, rollout управляет не теми Pods. Словарь labels должен быть ограниченным и иметь владельца: app, component, environment, team и version полезны только тогда, когда их значения согласованы между командами.

Annotations хранят неидентификационные метаданные и часто используются конкретными контроллерами. Не стоит переносить всю конфигурацию в annotations, особенно для современных сетевых API: Ingress annotations исторически делали поведение сильно зависящим от controller. Gateway API пытается перенести часть таких возможностей в типизированные ресурсы и роль-ориентированную модель. Это один из мотивов подробного сравнения Ingress и Gateway в томах 15–18.

## 6. Наблюдаемость архитектуры: что проверять первым

Для control plane наблюдают доступность API, задержку/ошибки запросов, состояние etcd, scheduler и controller-manager, сертификаты и общий quorum. Для node — kubelet, runtime, свободные ресурсы, давление диска/памяти, CNI agent и доступность DNS/dataplane. Для workload — desired versus available replicas, состояние Pod, probes, restart count и readiness endpoints. Метрика «все node Ready» полезна, но не доказывает, что приложение доступно через Service или Gateway.

Сквозной контроль должен использовать обычный путь пользователя. Например, тестовый Pod обращается к Service по DNS; внешний клиент идёт через выбранный Gateway/Ingress; контролируемая метрика проходит в наблюдаемость. Такая проверка обнаруживает ошибки, которые не видны у отдельных компонентов: неверный selector, CNI MTU, default-deny policy, TLS route или DNS. Важно не создавать уникальные labels/объекты на каждый тест, иначе сама проверка ухудшит кардинальность и состояние кластера.

Изменение control plane, CNI или ingress относится к data-plane/control-plane риску. До выпуска фиксируют версию, исходный статус, контрольные запросы, критерий остановки и rollback. После — проверяют несколько обычных интервалов, события и метрики компонентов. Этот метод будет повторяться во всех томах: Kubernetes сложен не количеством команд, а количеством границ, на которых один успешный локальный статус ещё не доказывает результат для пользователя.

## 7. API lifecycle: от HTTP-запроса до наблюдаемого состояния

kubectl — лишь один клиент Kubernetes API. Тот же API используют контроллеры, операторы, CI/CD, Terraform-провайдеры и другие программы. Запрос проходит аутентификацию: API server устанавливает, кто совершил действие. Затем авторизацию: RBAC и другие механизмы определяют, разрешён ли verb над конкретным ресурсом/namespace. После этого admission-цепочка может отклонить или изменить объект, а валидатор схемы проверяет допустимость полей. Только затем API server фиксирует объект в etcd и возвращает результат клиенту.

Это объясняет разницу между 401, 403 и ошибкой валидации. 401 означает, что сервер не смог установить личность клиента; 403 — личность известна, но действие запрещено; ошибка schema/admission означает, что доступ есть, но объект нарушает правило. При диагностике нельзя обходить эту цепочку выдачей cluster-admin «ради проверки»: так можно скрыть дефект ServiceAccount, RoleBinding или policy и расширить права production-клиента. Тест должен выполняться от имени реального потребителя.

У объекта есть spec и status. spec — желаемое состояние, которое объявляет пользователь; status заполняют контроллеры как наблюдаемое состояние. Status не следует редактировать вручную: его смысл принадлежит контроллеру конкретного API. Важнее читать conditions: они часто содержат reason и message, объясняющие, почему Deployment не доступен, Gateway не запрограммирован или PVC не привязан. Условие Ready без понимания владельца и generation не равно общему «всё работает».

## 8. Watches, resourceVersion и согласование контроллеров

Контроллеры не сканируют кластер с нуля каждую секунду. Они list-ят объекты, получают resourceVersion и затем watch-ят изменения. Когда desired state или зависимый объект меняется, контроллер кладёт ключ в рабочую очередь и выполняет reconcile: читает текущее состояние, сравнивает его с ожидаемым и делает минимальное изменение. Reconcile должен быть идемпотентным: повтор одного и того же события не должен создавать бесконечные побочные эффекты. Именно поэтому Kubernetes хорошо переносит рестарты контроллеров и повторные события.

Eventual consistency означает, что после apply не все потомки появляются мгновенно и не все status обновляются одновременно. API server подтвердил объект, затем контроллер увидел изменение, создал ReplicaSet, scheduler назначил Pod, kubelet подготовил его. Панель/скрипт, который проверяет состояние слишком рано, может ошибочно объявить сбой. Правильная приёмка использует status conditions, observedGeneration, timeout и события, а не фиксированную sleep-команду.

Рестарт контроллера обычно не должен ломать desired state, но может задержать согласование. Если контроллер создаёт внешние ресурсы — балансировщик, DNS, volume — важны его собственные retries, права и идемпотентность. В production следят за ошибками reconcile и задержкой между generation и observedGeneration. Это особенно понадобится для Gateway API и CNI-операторов: CR может существовать в API, но его data plane ещё не запрограммирован.

## 9. Admission: место, где платформа задаёт границы

Admission controller работает после авторизации и до сохранения объекта. Mutating admission может добавлять/изменять поля, например sidecar или default values; validating admission принимает либо отклоняет объект по политике. Webhook — внешний сервис, который участвует в этой критичной цепочке. Его недоступность, неверный TLS, слишком широкий match или режим failurePolicy могут либо остановить развёртывания, либо пропустить то, что платформа хотела запретить. Webhook — production-зависимость, а не вспомогательный Pod.

Политика полезна, когда формулирует проверяемое ограничение: контейнер не запускается privileged, image берётся из разрешённого registry, ресурс имеет requests/limits, Ingress/Gateway привязан к допустимому классу. Она не должна заменять техническое понимание приложения. Например, автоматическое добавление default requests удобно, но скрывает владельцу реальную потребность; лучше показывать отклонение и объяснять требование там, где это возможно.

При ошибке admission смотрят полный ответ API, событие и логи webhook, затем проверяют CABundle, service/endpoints, DNS/NetworkPolicy и RBAC webhook. Отключать webhook или менять failurePolicy на Ignore во время инцидента — рискованная операция: она меняет модель безопасности всех последующих объектов. Сначала определяют область сбоя, сохраняют конфигурацию и выбирают контролируемый откат.

## 10. Control plane HA и границы его доступности

Высокая доступность control plane означает, что API и хранилище состояния переживают отказ отдельных машин в рассчитанной topology. Обычно API server работает на нескольких узлах за стабильной точкой доступа, а etcd имеет нечётное число членов для quorum. Однако несколько виртуальных машин на одном физическом хосте, одном storage или одной сети не являются независимыми доменами отказа. Путь клиента к API, сертификаты, DNS/VIP и внешняя балансировка тоже входят в отказную модель.

etcd требует особенно аккуратной эксплуатации. Нельзя лечить проблему quorum удалением членов или очисткой данных наугад: можно превратить частичную недоступность в потерю состояния. У команды должен быть свежий снимок, проверенная процедура restore в изолированном контуре, известная версия Kubernetes/etcd и критерии, когда требуется эскалация. Snapshot без тестового восстановления не доказывает RPO; работающий API после рестарта не доказывает, что исторические объекты и secrets доступны корректно.

При недоступности API уже запущенные Pod могут продолжать работать на node, но управление деградирует: новые Pods не планируются, status/secret/config обновления не приходят, контроллеры не согласуют state. Это важное различие для коммуникации инцидента: «приложение отвечает» и «кластер управляем» — разные свойства. Наблюдаемость должна показывать API latency/errors, etcd health, quorum, controller/scheduler состояние и сквозной kubectl/API запрос от разрешённого клиента.

## 11. Системный алгоритм диагностики Kubernetes

Начните с вопроса и объекта. Например: «Deployment payments не имеет готовых реплик с 10:15 UTC». Проверьте desired/available replicas, conditions и events Deployment/ReplicaSet/Pod. Если Pod Pending — смотрите scheduler reason, requests, taints, affinity, quotas и node; если ContainerCreating — volumes, CNI, image/runtime; если Running, но не Ready — probes, приложение, DNS/зависимости. Такой переход от верхнего ресурса к следующей границе не даёт тратить время на не относящиеся команды.

Собирайте факты до изменения: generation, observedGeneration, node, image digest, последние events, restart count, status conditions, время ошибки. Затем меняйте одну переменную и повторяйте тот же контроль. Массовый rollout restart, ручное удаление всех Pods или выдача широких прав могут временно убрать симптом, но уничтожить причинную цепочку. Kubernetes предоставляет много автоматизации, поэтому особенно важно не мешать контроллерам во время исследования без ясного плана.

Завершение инцидента требует проверки пользовательского результата: Pod готов, Service имеет endpoints, соединение/запрос проходит, нужные метрики и alert вернулись в норму. Затем фиксируют причину, защиту и runbook. Если причиной был selector, policy, admission или quota, тест добавляют в pipeline/приёмку; иначе тот же дефект появится после следующей декларации. Этот метод будет применён далее к сети, CNI, Gateway и production-изменениям.

## 12. Домашняя практика

- Разверните небольшой локальный кластер и получите список control-plane и node-компонентов.

- Создайте Deployment с двумя репликами; проследите цепочку Deployment → ReplicaSet → Pod → назначенный node.

- Укажите selector Service, затем намеренно измените label Pod и проверьте EndpointSlice.

- Соберите события Pod в случаях Pending и ImagePullBackOff; для каждого назовите слой, который нужно исследовать.

- Зафиксируйте один сквозной проверочный маршрут: Pod → Service DNS → backend.
