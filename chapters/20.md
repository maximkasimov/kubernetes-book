# Kubernetes: подробный учебный материал — Наблюдаемость Kubernetes: метрики, логи, события и трассировка

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## Слои наблюдаемости

Наблюдаемость кластера включает control plane, nodes, kubelet/runtime, CNI, workloads и пользовательский путь. Метрика отвечает «сколько и как меняется», лог — «что произошло», trace — «как прошёл конкретный запрос», event — временная подсказка от Kubernetes. Один источник не заменяет другой.

## Состояние объектов

Собирайте conditions Deployment, Pod, Node, PVC, Gateway и их transition time. kube-state-metrics превращает состояние API-объектов в метрики, но не измеряет реальное использование ресурсов. Metrics Server нужен прежде всего для HPA/ kubectl top и не является системой долговременного мониторинга.

## Метрики ресурса

CPU, memory, filesystem, network и saturation node нужны вместе с requests/limits. Высокий CPU может означать полезную работу, throttling или нехватку лимита; OOM и eviction требуют отличать память контейнера от pressure node. Для CNI отслеживайте errors agent, IPAM utilisation, policy drops, conntrack и BGP/tunnel health.

## Логи и контекст

Собирайте stdout/stderr контейнеров, логи kubelet/runtime и CNI, сохраняя namespace, Pod, container, node, image и trace/request correlation. Не используйте label с request_id в метриках: это высокая кардинальность; для одиночного запроса используйте лог/trace. Политика retention и доступ к логам должны учитывать секреты и персональные данные.

## Alert и SLO

Alert должен соответствовать действию: недоступен API, нет ready replicas, исчерпан IPAM, Gateway не программируется, certificate истекает. Симптомные alert полезнее шума на каждый рестарт. SLO пользовательского сервиса связывают с error rate/latency и budget; инфраструктурная метрика нужна, когда предсказывает потерю этой цели.

Kubernetes monitoring: https://kubernetes.io/docs/tasks/debug/debug-cluster/resource-metrics-pipeline/ . kube-state-metrics: https://github.com/kubernetes/kube-state-metrics/tree/main/docs . Kubernetes logging: https://kubernetes.io/docs/concepts/cluster-administration/logging/.

## Официальные источники для темы

Сигнал alert должен вести к действию: «Gateway listener не готов», «Pod IP pool почти исчерпан», «нет готовых endpoints», «etcd/API errors растут». Метрика без порога, владельца и runbook создаёт шум. Отделяйте симптом для пользователя от причинной инфраструктурной метрики, чтобы on-call сначала восстанавливал сервис, а затем устранял корень.

## Углубление: alert design

Events имеют TTL и могут быть агрегированы, поэтому при инциденте сохраняйте их быстро, но не стройте compliance-аудит на events. Audit log API — отдельный поток, показывающий кто/что изменял в Kubernetes; его объём, чувствительность и retention проектируют заранее.

Логи приложения пишутся stdout/stderr контейнера, а kubelet/runtime/CNI/controller имеют свои журналы на nodes или в control plane. В централизованный backend добавляйте стабильный контекст namespace, Pod, container, node, workload и image digest. Не записывайте secrets, Authorization headers или персональные данные без политики redaction.

## Углубление: логи и события

Каждая метрика должна иметь владельца и вопрос: API latency — здоровье control plane, ready replicas — доступность workload, IPAM usage — способность CNI создавать Pods, Gateway programmed — готовность edge. Не добавляйте request_id/user_id как labels: для единичного расследования предназначены логи и traces.

Metrics Server агрегирует ресурсные метрики для kubectl top/HPA и не предназначен для долгой истории или точной диагностики. kube-state-metrics читает Kubernetes API и экспортирует состояние объектов; оно может отставать от реального процесса, но отлично отвечает на вопросы о desired/available replicas, conditions и scheduling. Node exporter/kubelet/cAdvisor отражают ресурсы node/container.

## Домашняя практика

В учебном кластере создайте минимальный пример рассматриваемой сущности. До apply выполните server-side dry-run и diff. После применения сохраните YAML, describe, events и логи. Внесите одну намеренную ошибку и объясните, какой компонент и какая condition показывают её причину. Production-секреты и данные в упражнении не используйте.
