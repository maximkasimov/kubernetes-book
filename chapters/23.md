# Kubernetes: подробный учебный материал — GitOps, Helm и поставки Kubernetes-приложений

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## Декларативный источник истины

GitOps хранит желаемое состояние в Git, а контроллер в кластере сравнивает его с API и согласует изменения. Это даёт аудит, воспроизводимость и drift detection, но секреты и bootstrap требуют отдельного дизайна. Не смешивайте ручное kubectl edit с GitOps без процедуры: следующий reconcile откатит ручное изменение.

Изучайте ресурс через Kubernetes API, а не как отдельный YAML-файл. API server хранит желаемое состояние, а контроллеры асинхронно приводят кластер к нему. Поэтому момент успешного apply и момент готовности приложения различны.

Для любого объекта различайте spec и status. spec задаёт намерение, status и conditions сообщают результат работы контроллеров. Conditions с полями type, status, reason и message обычно точнее краткой колонки kubectl get.

События (Events) полезны как временная диагностическая подсказка. Их нельзя считать долговременным аудитом: сохраняйте важные факты в системе логирования и runbook. Всегда проверяйте ownerReferences, чтобы понять, какой контроллер создал объект.

Безопасная диагностика идёт сверху вниз: пользовательский результат, ресурс верхнего уровня, его потомки, node/runtime и внешняя зависимость. Массовое удаление Pod или выдача cluster-admin может скрыть причину, а не устранить её.

## Helm

Chart параметризует шаблоны Kubernetes. Values должны иметь ясную схему и безопасные defaults; шаблон не должен скрывать критичные security/network поля. helm install/upgrade меняет объекты через release metadata, но не заменяет проверку состояния и пользовательского пути. Для сложных ресурсов важны ownership поля и совместимость CRD.

## Rollback

Откат — это новая декларация к известной версии, но rollback приложения может быть несовместим с миграцией данных. Сохраняйте версию chart, values, image digest, CRD и порядок миграций. После rollback проверяйте Gateway/Service, policy, data и синтетический пользовательский запрос.

## Официальные источники для темы

Промотируйте immutable image digest между окружениями, а не пересобирайте «тот же» тег. Pipeline проверяет источник, подпись/сканирование, rendered manifests, policy, diff и пользовательский smoke test. Identity CI/GitOps должна иметь минимальные права: компрометация repo не должна давать неограниченное изменение кластера.

## Углубление: promotion и provenance

CRD жизненный цикл требует особой осторожности: изменение schema может сделать существующие объекты невалидными, а rollback chart не всегда откатывает CRD/data. Версии CRD/controller и миграции тестируют как отдельный production change.

Helm рендерит шаблоны и хранит release state; он не заменяет schema validation и policy checking. values должны быть типизированы схемой, а chart — избегать неявных privileged/network defaults. Hooks способны выполнять Jobs вне обычного reconcile-порядка, поэтому должны иметь timeout, idempotency и наблюдаемость.

## Углубление: Helm ownership

Разделите bootstrap platform и application delivery. Сначала CRD/controllers/GatewayClass/StorageClass, затем ресурсы, которые на них ссылаются. Порядок важен: применение HTTPRoute до Gateway controller может быть принято API, но не запрограммировано в dataplane.

GitOps controller периодически или по webhook сравнивает Git desired state с API. Drift может быть следствием ручной правки, мутации admission, другого controller или поля, которое Git не должен владеть. Не включайте force/prune автоматически без классификации ресурсов: удаление внешнего LoadBalancer/PVC или CRD может иметь необратимый эффект.

## Домашняя практика

В учебном кластере создайте минимальный пример рассматриваемой сущности. До apply выполните server-side dry-run и diff. После применения сохраните YAML, describe, events и логи. Внесите одну намеренную ошибку и объясните, какой компонент и какая condition показывают её причину. Production-секреты и данные в упражнении не используйте.
