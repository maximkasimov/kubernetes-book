# Kubernetes: подробный учебный материал — Установка, обновление и высокая доступность Kubernetes

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## Дизайн кластера

До установки выберите CIDR Pod/Service без пересечений, topology control plane, container runtime, CNI, ingress/Gateway, storage и механизм доступа. Несколько VM на одном storage/host не создают независимые домены отказа. Документируйте DNS/VIP/load balancer, сертификаты, firewall и владельцев каждого слоя.

## Control plane и etcd

API servers работают за стабильной точкой доступа; etcd требует нечётного числа членов для quorum. Сбой API не обязательно останавливает уже работающие Pod, но запрещает новое планирование и согласование. Snapshot etcd без регулярного восстановления не доказывает способность восстановить кластер. Никогда не удаляйте членов etcd или данные ради «починки quorum» без проверенного runbook.

## Обновление

Планируйте upgrade по поддерживаемому version skew: control plane, nodes, kubelet, CNI, CSI и addons должны быть совместимы. Сначала изучите release notes и deprecated API, сделайте backup/restore-проверку, обновляйте control plane и nodes управляемыми волнами, проверяя API, scheduling, сеть и workload после каждого шага.

## Drain и workloads

cordon предотвращает новое размещение, drain выселяет управляемые Pods с учётом PDB и daemonsets. Drain не делает приложение автоматически HA: у него должны быть реплики, PDB, корректные readiness и capacity на оставшихся nodes. Stateful workload требует отдельной процедуры из-за quorum/data locality.

## Rollback и проверка

Rollback определяется до изменения: версия образов/манифестов, backup, точка восстановления и критерий stop. После обновления проверяйте не только kubectl get nodes, но CNI attach, DNS, Service, policy, CSI mount, Gateway/TLS и пользовательские синтетические запросы.

Highly available clusters: https://kubernetes.io/docs/setup/production-environment/tools/kubeadm/high-availability/ . Version skew: https://kubernetes.io/releases/version-skew-policy/ . Upgrading kubeadm clusters: https://kubernetes.io/docs/tasks/administer-cluster/kubeadm/kubeadm-upgrade/.

## Официальные источники для темы

Проверяйте API и nodes, создание нового Pod, DNS, Service, NetworkPolicy, CSI mount, Gateway listener/Route/TLS и прикладной synthetic transaction. Здоровые старые Pods могут скрыть дефект, который проявится только при первом новом deployment. Поэтому тест создания/удаления тестового Pod — обязательная часть приёмки CNI/runtime.

## Углубление: приемка после изменения

Draining node учитывает PDB, но не гарантирует сохранность singleton/stateful workload. Для таких систем создайте отдельный runbook: лидерство, quorum, backup, volume detach/attach и разрешённое окно. Не совмещайте массовый app rollout, CNI update и Kubernetes version upgrade в одном изменении.

Перед обновлением изучите поддерживаемую версионную матрицу и API removals. Сначала совместимый control plane, затем workers контролируемыми волнами, с проверкой CNI/CSI/Gateway/addons после каждой. kubelet не должен перескакивать через неподдерживаемые версии. Образы компонентов и registry должны быть доступны до окна работ.

## Углубление: upgrade sequence

etcd требует quorum: для трёх членов требуется два доступных. Latency/disk I/O etcd напрямую влияет на API. Snapshot делайте регулярно, защищайте шифрованием и проверяйте restore на отдельном контуре с подходящей версией etcd/Kubernetes.

Три control-plane VM не дают высокой доступности, если они зависят от одного physical host, SAN, rack, power feed или L2 domain. Нарисуйте путь клиента к API: DNS/VIP/load balancer, API servers, etcd, сеть и PKI. Для каждого сегмента назовите домен отказа и способ проверки failover.

## Домашняя практика

В учебном кластере создайте минимальный пример рассматриваемой сущности. До apply выполните server-side dry-run и diff. После применения сохраните YAML, describe, events и логи. Внесите одну намеренную ошибку и объясните, какой компонент и какая condition показывают её причину. Production-секреты и данные в упражнении не используйте.
