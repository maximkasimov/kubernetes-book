# Kubernetes: подробный учебный материал — Ingress: традиционная модель HTTP-маршрутизации и её границы

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## Ingress и Ingress Controller

Ingress — API-объект для HTTP/HTTPS маршрутов host/path к Service. Сам по себе он не открывает порт и не проксирует трафик: это делает Ingress Controller, который наблюдает объект и программирует свой dataplane. IngressClass связывает ресурс с конкретной реализацией. Несовпадающий class даёт YAML без работающего маршрута.

## Путь запроса

Клиент приходит на внешний LoadBalancer/NodePort controller, TLS может завершиться там, controller выбирает правило host/path и отправляет запрос в Service, затем в endpoint Pod. На каждом шаге отдельная причина отказа: DNS/балансировщик, listener/controller, правило, Service/EndpointSlice, NetworkPolicy или readiness backend.

## TLS

tls.secretName обычно ссылается на Secret того же namespace, содержащий certificate/key. Controller может поддерживать дополнительные аннотации/сертификатные интеграции, но это vendor-specific. Проверяйте SNI, имя в сертификате, цепочку и факт загрузки controller, а не только существование Secret.

## Ограничения API

Ingress исторически фокусируется на HTTP и оставляет расширенные возможности аннотациям конкретного controller: retries, canary, header matching, traffic splitting, TCP/UDP. Это ухудшает переносимость и управляемость. Kubernetes объявил Ingress frozen: он поддерживается, но новые возможности развивают Gateway API.

## Безопасная эксплуатация

Не смешивайте public и internal маршруты в одном классе без явной модели. Ограничивайте допустимые annotations admission-политикой, задавайте timeouts и лимиты на стороне controller, отслеживайте 4xx/5xx/latency. Ingress controller — shared edge: его обновление и capacity влияют на много команд.

Kubernetes Ingress: https://kubernetes.io/docs/concepts/services-networking/ingress/ . Ingress is frozen and Gateway API: https://kubernetes.io/docs/concepts/services-networking/ingress/#note.

## Официальные источники для темы

Новые функции Ingress часто выражались аннотациями, поэтому один YAML терял переносимость и смешивал platform-настройки с маршрутом приложения. Gateway API вводит GatewayClass/Gateway/Route и статус каждого уровня. Ingress остаётся поддерживаемым, но frozen: не планируйте новую платформенную возможность только на его расширениях, если Gateway API подходит.

## Углубление: почему переходят на Gateway

Публичный путь включает DNS, внешний балансировщик, listener proxy, TLS, rule, Service и endpoint. Для каждого слоя определите health check и логи. Ошибка TLS secret, переполненный proxy или неверный IngressClass не будет видна из логов backend Pod.

Ingress API намеренно не определяет реализацию proxy. NGINX Ingress, Traefik, HAProxy и облачные контроллеры могут по-разному интерпретировать annotations, path matching, TLS и source IP. Перед изменением установите controller/version/class и его документацию. Объект Ingress без работающего controller — только запись в API.
