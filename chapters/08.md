# Kubernetes: подробный учебный материал — Сетевая модель Kubernetes: адресация, DNS и путь пакета

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## Сетевой контракт

Kubernetes ожидает связность Pod-to-Pod без NAT в базовой модели, доступ node к Pod и Pod к node. CNI реализует контракт: создаёт network namespace, интерфейс, IP, маршруты и dataplane. CIDR Pod не должен пересекаться с node/корпоративной сетью, иначе маршрутизация станет неоднозначной.

Изучайте ресурс через Kubernetes API, а не как отдельный YAML-файл. API server хранит желаемое состояние, а контроллеры асинхронно приводят кластер к нему. Поэтому момент успешного apply и момент готовности приложения различны.

Для любого объекта различайте spec и status. spec задаёт намерение, status и conditions сообщают результат работы контроллеров. Conditions с полями type, status, reason и message обычно точнее краткой колонки kubectl get.

События (Events) полезны как временная диагностическая подсказка. Их нельзя считать долговременным аудитом: сохраняйте важные факты в системе логирования и runbook. Всегда проверяйте ownerReferences, чтобы понять, какой контроллер создал объект.

Безопасная диагностика идёт сверху вниз: пользовательский результат, ресурс верхнего уровня, его потомки, node/runtime и внешняя зависимость. Массовое удаление Pod или выдача cluster-admin может скрыть причину, а не устранить её.

## Путь Pod-to-Pod

На одной node пакет проходит veth и dataplane плагина. Между nodes путь может быть L3-маршрутизацией, оверлеем с инкапсуляцией либо eBPF. Выбор влияет на MTU, source IP, firewall и диагностику. Внешний исходящий трафик может SNAT-иться node/gateway; ingress source IP и egress source IP — отдельные свойства.

## Service и EndpointSlice

Service даёт стабильный виртуальный адрес и selector либо вручную созданные EndpointSlice. kube-proxy или eBPF выбирает готовый backend. ClusterIP внутренний, NodePort открывает node-порт, LoadBalancer требует внешней интеграции. DNS-ответ Service не доказывает доступность: проверяйте selector, EndpointSlice, targetPort, readiness, policy и dataplane.

## DNS

## MTU и диагностика

Оверлей тратит MTU на заголовки. При несоответствии маленький запрос может пройти, а TLS/крупный ответ зависнуть. Начинайте с исходного Pod/IP, получателя/порта, DNS, первой точки отказа и владельца маршрута/policy. Тест с node не заменяет тест из Pod того же namespace.

## Официальные источники для темы

kube-proxy в iptables/IPVS либо CNI eBPF dataplane реализует Service virtual IP и выбор backend. У контроллера Service, EndpointSlice и node dataplane разные роли: изменения selector сначала появляются в EndpointSlice, затем должны быть применены на nodes. Частичная задержка/ошибка на node объясняет, почему один клиент видит новый backend, а другой нет.

## Углубление: сервисный dataplane

DNS-имя Service стабильно в пределах cluster domain, но не проверяет прикладную готовность. Клиент должен иметь timeout/retry/connection policy, а Service — readiness/endpoints. Эти две стороны вместе определяют устойчивость discovery.

CoreDNS получает запросы через Service. Если DNS Pod не ready, Service endpoints пусты или NetworkPolicy закрыла UDP/TCP 53, приложения получают временные resolution errors. Проверяйте и UDP, и TCP fallback, search domains, ndots и влияние большого числа коротких DNS-запросов. Не фиксируйте Pod IP в конфигурации как обход DNS: это ломает lifecycle workload.

## Углубление: DNS и discovery

Kubernetes API не требует, чтобы Service IP был маршрутизируемым за пределами кластера; dataplane перехватывает его на nodes. Поэтому попытка обратиться к ClusterIP с внешней машины обычно не является корректной проверкой Service.

Обычно различают node network, Pod network и Service network. Node IP нужен для управления/межузлового транспорта, Pod IP — для конечных workload, Service ClusterIP — виртуальная точка выбора backend. Эти диапазоны не должны пересекаться между собой и с reachable внешними сетями. Ошибка CIDR нередко проявляется после подключения VPN/peering, а не при первом локальном тесте.

## Домашняя практика

В учебном кластере создайте минимальный пример рассматриваемой сущности. До apply выполните server-side dry-run и diff. После применения сохраните YAML, describe, events и логи. Внесите одну намеренную ошибку и объясните, какой компонент и какая condition показывают её причину. Production-секреты и данные в упражнении не используйте.
