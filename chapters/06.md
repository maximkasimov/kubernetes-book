# Kubernetes: подробный учебный материал — Планирование и ресурсы: как Kubernetes выбирает node и защищает кластер

Полная теоретическая лекция для инженера, который начинает работать с Kubernetes. Термины на английском приведены в скобках там, где они используются в API или официальной документации.

## Решение scheduler

Scheduler фильтрует nodes по requests и allocatable, nodeSelector/affinity, taints/tolerations, volumes и topology rules, затем оценивает подходящие варианты. Он опирается на requests, а не на моментное потребление. Pending диагностируют по PodScheduled и FailedScheduling, не удалением Pod.

Изучайте ресурс через Kubernetes API, а не как отдельный YAML-файл. API server хранит желаемое состояние, а контроллеры асинхронно приводят кластер к нему. Поэтому момент успешного apply и момент готовности приложения различны.

Для любого объекта различайте spec и status. spec задаёт намерение, status и conditions сообщают результат работы контроллеров. Conditions с полями type, status, reason и message обычно точнее краткой колонки kubectl get.

События (Events) полезны как временная диагностическая подсказка. Их нельзя считать долговременным аудитом: сохраняйте важные факты в системе логирования и runbook. Всегда проверяйте ownerReferences, чтобы понять, какой контроллер создал объект.

Безопасная диагностика идёт сверху вниз: пользовательский результат, ресурс верхнего уровня, его потомки, node/runtime и внешняя зависимость. Массовое удаление Pod или выдача cluster-admin может скрыть причину, а не устранить её.

## CPU, память и QoS

CPU request участвует в размещении и распределении времени, limit ограничивает cgroup. Memory limit — жёсткая граница: превышение может закончиться OOMKilled. Memory request влияет на размещение и QoS. Guaranteed требует равных request/limit CPU и памяти для каждого контейнера; Burstable имеет requests; BestEffort их не имеет. Это влияет на eviction, но не отменяет capacity planning.

## Taints и affinity

Taint отталкивает Pod без toleration. NoSchedule запрещает новое размещение, NoExecute также выселяет работающие Pods. Toleration разрешает размещение, но не требует его: требование задают selector/affinity. Pod anti-affinity и topologySpreadConstraints распределяют реплики, но чрезмерная строгость создаёт Pending в малом кластере.

## Quota и priority

ResourceQuota ограничивает сумму ресурсов и объектов namespace; LimitRange задаёт defaults/min/max. Ошибка quota может проявиться на admission до scheduler. PriorityClass допускает preemption, но не создаёт capacity; высокие приоритеты резервируют для согласованно критичных компонентов.

## Углубление: affinity и topology

CPU throttling возникает при CPU limit: latency приложения может расти при умеренном фактическом CPU. Для чувствительных сервисов limit CPU выбирают после профилирования, а не механически. CPU request остаётся важен для fair sharing и scheduling.

OOMKilled означает, что процесс превысил доступную ему память/cgroup-предел, а eviction означает, что kubelet выселил Pod из-за pressure node. В обоих случаях проверьте request/limit, actual working set, node conditions и конкурирующие workload, но corrective action различается. Увеличение memory limit без измерения может просто перенести pressure на node.

## Углубление: eviction и OOM

Отдельно учитывайте maxPods и сетевую ёмкость CNI. Node может иметь CPU, но быть на лимите Pod IP, ENI/интерфейсов, volumes или ports. Capacity report должен включать эти дискретные ресурсы, иначе autoscaling не устраняет Pending.

Node status сообщает capacity и allocatable. Allocatable меньше физической ёмкости: часть резервируется для системы, kubelet и eviction thresholds. Scheduler сравнивает requests с allocatable, поэтому «на node есть свободная память» по top не отменяет Pending, если заявленные requests не помещаются.

В учебном кластере создайте минимальный пример рассматриваемой сущности. До apply выполните server-side dry-run и diff. После применения сохраните YAML, describe, events и логи.
